# Aithena v1.14.0 Release Notes — Testing & Infrastructure Hardening

_Date:_ 2026-03-26  
_Prepared by:_ Newt (Product Manager)

## Summary

Aithena v1.14.0 is a **maintenance and cleanup release** focused on testing coverage, infrastructure reliability, and dependency updates. This release does not ship user-facing features but hardens the existing v1.13.x surface through expanded reindex endpoint test coverage, benchmark script improvements for the e5-base embedding model, and critical infrastructure fixes for Solr bind-mount permissions and restore script behavior.

**Key Achievement:** The previous v1.14.0 milestone (A/B evaluation UI for comparing embedding models) was explicitly cancelled in favor of shipping e5-base as the standard model. v1.14.0 replaces that milestone with hardening work, closing out the A/B evaluation framework and consolidating around the proven e5-base model.

### What's Delivered in v1.14.0

- ✅ **Reindex Endpoint Test Coverage** — 13 new tests for the admin reindex feature (POST `/v1/admin/reindex`)
- ✅ **Benchmark Script Updates** — Refactored for single-collection e5-base setup, removing A/B scaffolding
- ✅ **Solr Bind-Mount Permissions** — Fixed volume permission issues via root entrypoint + gosu privilege drop
- ✅ **Restore Script DRY_RUN Compliance** — Drill restore scripts now respect `DRY_RUN` environment variable in pre-flight checks
- ✅ **Dependency Upgrades** — @eslint/js v10, actions/setup-node v6.3.0
- ✅ **Lint & Code Quality** — Admin tab handler cleanup, Squad heartbeat detection refinement, Dependabot triage permission fixes

### Infrastructure Track: Solr Bind-Mount Permissions

#### Issue: Solr Container Cannot Write to Bind-Mounted Volume

**Problem:** Solr container runs as non-root user `solr` (UID 8983). When Solr data directory is bind-mounted from the host (e.g., `/host/data:/var/solr/data`), the container often lacks write permission if the host directory is owned by root or a different user.

**Solution:**
- Modified Solr container to run entrypoint as root
- Uses `gosu` to drop privileges to unprivileged `solr` user before starting JVM
- Directory ownership corrected on startup
- File permissions set to allow `solr:solr` to write to volume

**Files:**
- `src/solr/entrypoint-sasl.sh` — Modified to handle root startup + privilege drop
- `docker-compose.yml` — Entrypoint configuration updated
- `docker-compose.prod.yml` — Same fixes applied for production

**Impact:** Bind-mounted volumes work reliably; operators can use persistent storage for Solr without permission errors.

---

### Infrastructure Track: Restore Script DRY_RUN Compliance

#### Issue: Drill Restore Scripts Ignore DRY_RUN in Pre-Flight Checks

**Problem:** Restore scripts (used to bootstrap drill/Solr data from offline bundles) did not respect the `DRY_RUN` environment variable during pre-flight checks. Scripts would validate and prepare data even when `DRY_RUN=true`, potentially modifying state in dry-run mode.

**Solution:**
- Updated restore script logic to check `DRY_RUN` before any state-modifying operations
- Pre-flight checks now skip file I/O when `DRY_RUN=true`
- Dry-run mode reports what would happen without executing
- Allows operators to validate restore process before committing

**Files:**
- `scripts/restore.sh`, `scripts/restore-critical.sh`, `scripts/restore-high.sh`, `scripts/restore-medium.sh` — Updated with DRY_RUN pre-flight checks
- Admin manual updated with DRY_RUN usage examples

**Impact:** Offline restoration is safer; operators can test restore procedures without side effects.

### Benchmark Script Refactoring

#### Issue: Benchmark Scripts Still Reference A/B Setup

**Problem:** Benchmark scripts contained scaffolding for A/B testing (dual embedding models, dual indexers, comparison metrics). This was removed from v1.14.0 scope as A/B evaluation UI was explicitly cancelled.

**Solution:**
- Refactored benchmarks for single-collection, single-model (e5-base) setup
- Removed dual-model comparison logic
- Simplified to measure e5-base performance and relevance
- Benchmarks now focus on:
  - Indexing throughput (documents/second)
  - Query latency (milliseconds)
  - Vector similarity accuracy (nDCG, MRR)
  - Memory usage under load

**Files:**
- `scripts/benchmark/run_benchmark.py` — Benchmark runner for e5-base performance testing
- `scripts/benchmark/queries.json` — Query suite for relevance evaluation
- `scripts/benchmark/README.md` — Benchmark documentation updated

**Impact:** Benchmarks are cleaner and more maintainable; focus is on production e5-base performance, not model comparison.

## A/B Evaluation Framework Closure

As documented in `.squad/decisions.md`, the v1.14.0 milestone originally planned to ship an A/B evaluation UI for end-users to compare distiluse-base and e5-base embeddings side-by-side. **This feature was explicitly cancelled** in favor of shipping e5-base as the standard model.

**Rationale:** Internal benchmarks demonstrated e5-base superiority across all metrics (relevance, speed, memory). End-user evaluation was unnecessary.

**Impact on Release:**
- All A/B evaluation issues (#900–918) closed as "not planned"
- A/B testing infrastructure remains in codebase but unused (gated behind `ENABLE_AB_TEST=false`)
- v1.14.0 replaces A/B work with infrastructure hardening (reindex tests, Solr permissions, restore scripts)
- **Users are NOT affected**—the A/B UI was never shipped and is not visible

## Deployment Notes

### For Deployments Running v1.13.x

v1.14.0 includes critical fixes for Solr bind-mount permissions and restore script DRY_RUN compliance. If you are using bind-mounted volumes or offline restore workflows:

```bash
# 1. Fetch the new release
git fetch origin
git checkout v1.14.0

# 2. Update Docker images
docker compose pull

# 3. Restart services with new entrypoint
docker compose down
docker compose up -d

# 4. Verify services are healthy
docker compose ps
sleep 30 && docker compose logs | grep -i error
```

### For New Deployments

Install v1.14.0 directly. No special steps needed — bind-mount permissions and restore script behavior are correct by default.

### Offline Deployment (Restore Scripts)

If using offline deployment with restore scripts:

```bash
# Test on air-gapped host, dry-run before committing
DRY_RUN=true ./scripts/restore.sh
# Review output
DRY_RUN=false ./scripts/restore.sh
```

## Validation Checklist

Before declaring v1.14.0 production-ready:

### Services & Health
- [ ] All services start: `docker compose up -d`
- [ ] All services report healthy: `docker compose ps`
- [ ] No errors in startup logs: `docker compose logs | grep -i error`

### Solr Bind-Mount Permissions
- [ ] Solr data directory is writable: `docker compose exec solr ls -la /var/solr/data`

### Reindex Feature
- [ ] Admin reindex endpoint responds: `curl -u admin:password http://localhost:8080/v1/admin/reindex?dry_run=true`
- [ ] Reindex returns proper status (200 for success, 4xx for errors)
- [ ] DRY_RUN mode doesn't modify state (verify with dry-run test)

- [ ] Collections feature functions (create, edit, add books)
- [ ] Results ranked by relevance

### Offline Deployment (if applicable)
- [ ] Offline bundle exports successfully
- [ ] Restore script respects DRY_RUN: `DRY_RUN=true ./restore.sh` produces no side effects
