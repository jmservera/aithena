### 1. Folder Facets (PRD-facet) — #1176, #1177, #1178, #1179, #1202, #1203, #1204, #1205

A new hierarchical folder facet system enables users to filter search results by document location in the library structure.

- Shows folder names with document counts per level
- Supports click-to-filter interactions

**#1178 / #1204:** Add folder filter integration to search UI
- Clicking a folder node filters results to that folder and its children (recursive)
- Selected folders appear in the active filters panel
- Clear all and individual filter removal work as expected

**#1179 / #1205:** Add comprehensive folder facet tests
- Backend API tests for folder facet extraction and filtering
- Frontend component tests for tree rendering and click interactions
- Integration tests for end-to-end folder filtering workflows

#### User Impact

Users can now organize and navigate searches by folder structure, making it easier to narrow results to specific library sections (e.g., "all books in English/Philosophy").

### 2. Backup & Disaster Recovery (PRD-bcdr) — #1180, #1181, #1182, #1183, #1184, #1185, #1186, #1206, #1207, #1208, #1209, #1210, #1211, #1212

A multi-tiered backup and restore system with orchestration scripts, admin API endpoints, and operator dashboards.

#### Backup Architecture

**Tier 1 — Critical (High frequency, restore priority):**
- Authentication database (`/data/auth/users.db`)
- Collections database (`/data/collections/`)
- Secrets management store (JWT secret, credentials)

**Tier 2 — High (Medium frequency):**
- Solr indices and collection configurations
- ZooKeeper coordination data

**Tier 3 — Medium (Lower frequency, optional):**
- Redis cache state
- RabbitMQ queue persistence

#### New Components

**#1180–#1183:** Backup orchestrator script
- Schedules backups according to tier priority and frequency
- Runs all tiers in sequence with error handling and logging
- Stores backups in versioned directories with timestamps
- Supports dry-run mode for validation

**#1185–#1186:** Restore orchestrator script
- Loads and lists available backups
- Supports dry-run preview before executing restore
- Verifies backup integrity before restore
- Restores tiers in reverse priority order
- Post-restore health checks

**#1206–#1209:** Backup admin API endpoints
- `POST /v1/admin/backup/now` — trigger on-demand backup
- `GET /v1/admin/backup/status` — current backup state
- `GET /v1/admin/backup/history` — list available backups
- `POST /v1/admin/backup/restore` — initiate restore with preview

**#1210–#1212:** React backup dashboard
- Visual backup status card in the admin portal
- Tier health indicators
- On-demand backup trigger button
- Backup history table with restore entry points
- Restore wizard with select/preview/confirm/progress steps

#### Documentation

New `docs/guides/disaster-recovery-runbook.md`:
- Monthly restore drill procedures
- Recovery time objectives (RTO) for each tier
- Capacity planning for backup storage
- Archive rotation and retention policies
- Post-restore validation checklist

#### Operator Impact

Operators gain a comprehensive, tested disaster recovery capability with monthly drill procedures and visual dashboards for monitoring backup health and triggering manual backups.

### 5. Bug Fixes

#### #1234: PDF Embedded Viewer Fix

**Issue:** Some PDFs were not loading in the embedded viewer. Browser console showed `X-Frame-Options` header violations.

**Root Cause:** NGINX was returning `X-Frame-Options: deny` on the `/documents/` location, preventing iframe embedding of PDFs.

**Fix:** Changed X-Frame-Options to `SAMEORIGIN` on the `/documents/` NGINX location, allowing PDFs served from the same origin to be embedded.

**Workaround (pre-v1.18.0):** Users could click "Open in new window" button to view the PDF.

**User Impact:** PDF embedded viewer now works reliably for all PDFs. Workaround no longer needed.

#### #1233: Collections Book Card Consistency

**Issue:** Collections displayed books in a different card format than the Library, breaking UI consistency.

**Root Cause:** Collections were using custom card templates instead of reusing the shared book card components.

**Fix:** Updated collections UI to display books using the same card/list components as the Library (title, author, Open PDF button), while retaining collection-specific actions (notes, remove from collection).

**User Impact:** Collections now match the visual style and behavior of the Library, creating a consistent reading experience throughout the application.

### 6. Security

#### #1237: GitHub Actions Secrets Protection (zizmor)

**Issue:** GitHub code scanning alert #230 flagged secrets potentially accessible outside a dedicated environment in `build-containers.yml` CI workflow.

**Root Cause:** Build workflow was accessing secrets (Docker registry credentials) without explicit environment protection rules.

**Fix:** 
- Created dedicated GitHub environment for build-containers.yml with protection rules
- Updated workflow to gate secret access to the protected environment
- Reduced exposure surface for credentials

**User Impact:** Improved security posture; no functional changes for deployments.
