# Aithena v1.13.0 Release Notes — Offline Deployment & Infrastructure Hardening

_Date:_ 2026-03-25  
_Prepared by:_ Newt (Product Manager)

## Summary

Aithena v1.13.0 is a **feature and security release** delivering three major themes: **air-gapped offline deployment**, **improved search ergonomics**, and **comprehensive infrastructure security hardening**. The release closed 10 milestone issues, enabling organizations to deploy Aithena in fully disconnected environments and hardening all services against unauthorized access.

### Key Deliverables

- ✅ **Offline Installer & Export** — Air-gapped deployment without external network access; offline export + verify + install flow
- ✅ **Diacritic-Insensitive Search** — Default behavior with configurable sensitivity; simple query-language tips
- ✅ **Security Hardening** — Nginx HSTS headers, Solr auth/authz, ZooKeeper SASL+ACLs, Redis ACLs, RabbitMQ per-service credentials, non-root containers

## Shipped Changes

### Feature Track: Offline Deployment

#### Issue: Air-Gapped Offline Installer Package

**Problem:** Aithena required internet connectivity during deployment (Docker image pulls, package installations, third-party API calls). Organizations with fully disconnected environments or strict air-gap policies could not deploy Aithena.

**Solution:**
- New offline installer package that bundles all Docker images, Python dependencies, and auxiliary files
- Offline export workflow: `./scripts/export-images.sh` (runs on networked machine with internet access)
- Offline deployment workflow: Extract package and run `./install.sh` (runs on air-gapped host)
- Verification tool: `./verify.sh` validates service health after installation
- No external network access required after bundle import
- All configurations and secrets generated on the target host (not pre-bundled)

**User Impact:**
- Regulated environments (government, healthcare, classified networks) can now deploy Aithena
- Offline deployments follow the same security practices as online deployments
- Operators export bundles from a networked staging machine and transfer via approved media (USB, air-gap appliances)

**Deployment Checklist:**
- [ ] On networked machine: `./scripts/export-images.sh` to create `staging/aithena-offline-v*.tar.gz`
- [ ] Transfer bundle to air-gapped environment via approved media
- [ ] Extract: `tar xzf aithena-offline-v*.tar.gz && cd aithena-offline-v*/`
- [ ] Run installer: `sudo ./install.sh`
- [ ] Configure credentials in `.env` at installation directory
- [ ] Verify services: `./verify.sh` or `docker compose ps`

### Feature Track: Search Enhancements

#### Issue: Add Diacritic-Insensitive Search by Default (but Configurable)

**Problem:** Search for "café" returned no results if documents contained "cafe" (or vice versa). Users unfamiliar with diacritics lost results due to accenting mismatches. No way to control this behavior.

**Solution:**
- Diacritic-insensitive search **enabled by default** via Solr analyzer chain (ASCII folding)
- Solr query analyzer removes accents: "café" matches "cafe", "naïve" matches "naive"
- Configuration flag `SOLR_ASCII_FOLDING` (default: true) to control behavior
- Admin manual documents the configuration option

**Impact on Users:**
- Broader search results by default for queries with accented characters
- Improved discoverability for international and multilingual libraries
- Transparent behavior explained in admin and user documentation

**Verification:**
- [ ] Search for "cafe" — results include documents with "café"
- [ ] Search for "naïve" — results include documents with "naive"
- [ ] Admin manual documents diacritic behavior
- [ ] User manual explains diacritic-insensitive search

### Security Track: Infrastructure Hardening

#### Issue: Nginx Missing HSTS Header for TLS Deployments (N-1)

**Problem:** nginx reverse proxy did not include `Strict-Transport-Security` headers. Deployments using TLS were vulnerable to downgrade attacks.

**Solution:**
- HSTS header configuration defined for nginx in `docker-compose.ssl.yml`
- Policy: `Strict-Transport-Security: max-age=31536000; includeSubDomains` (1 year, includes subdomains)
- Operators enable via `docker-compose.ssl.yml` overlay for HTTPS deployments
- Header sent only when TLS is active

**Impact on Operators:**
- TLS-enabled deployments can enforce HTTPS and prevent downgrade attacks
- Browser enforces HTTPS for future requests to the domain
- Reduces risk of session hijacking via MitM attacks

---

#### Issue: RabbitMQ Uses Single Shared User for All Services (MQ-1)

**Problem:** All services (document-indexer, document-lister) used the same hardcoded RabbitMQ username/password. A compromised service could access queues belonging to other services.

**Solution:**
- Created per-service RabbitMQ credentials in `docker-compose.yml`
- Each service now has its own username and password (e.g., `indexer_user`, `lister_user`)
- Per-service queue access enforced at RabbitMQ level
- RabbitMQ user permissions scoped to specific exchanges and queues
- Credentials injected as environment variables; no hardcoding in code

**Impact on Operators:**
- Service credential compromise is isolated (attacker gains access only to that service's queues)
- Credential rotation per-service without affecting other services
- Credentials stored in `.env` (generated by installer)

---

#### Issue: ZooKeeper AdminServer Exposed on Port 8080 (Z-3)

**Problem:** ZooKeeper AdminServer listened on port 8080 without authentication, exposing cluster management endpoints to the internal Docker network.

**Solution:**
- Disabled AdminServer by default (via Java system property `-Dzookeeper.admin.enableServer=false`)
- AdminServer not exposed in `docker-compose.yml`
- If operators need admin functionality, they must explicitly enable it and restrict network access

**Impact on Operators:**
- Reduces attack surface for unauthorized cluster manipulation
- No unintended exposure of ZooKeeper internals

---

#### Issue: 4/6 Custom Containers Run as Root (D-2)

**Problem:** Dockerfiles for `document-indexer`, `document-lister`, `solr-search`, and `admin` ran processes as `root` (UID 0). A container breakout could grant full host compromise.

**Solution:**
- Added non-root user to all custom service Dockerfiles
- Each container now runs as a dedicated user (e.g., `indexer`, `lister`, `solr-search`, `admin`) with UID ≥ 1000
- File ownership and permissions updated to match new user
- Official images (`solr`, `redis`, `rabbitmq`, `zookeeper`) already run as non-root

**Impact on Operators:**
- Container breakout risks reduced (attacker is restricted to non-root privileges)
- Compliant with security best practices and container security standards

---

#### Issue: Redis Has No ACLs — Dangerous Commands Available to All Clients (R-1, R-2)

**Problem:** Redis ran with default configuration allowing any client to execute dangerous commands (FLUSHDB, FLUSHALL, CONFIG SET). Internal Docker network access was sufficient for attack.

**Solution:**
- Enabled Redis ACL (Access Control Lists) with default user restrictions
- Created dedicated Redis user for indexing services (`indexer_user`)
- Restricted `indexer_user` to safe commands only: GET, SET, DEL, LPUSH, RPOP, LRANGE, SCAN, MGET (and variations)
- Blocked dangerous commands: FLUSHDB, FLUSHALL, CONFIG, SHUTDOWN, SLAVEOF
- Admin credentials for local debugging (accessible only via Unix socket if needed)
- Credentials injected via `.env`

**Impact on Operators:**
- Unauthorized clients cannot flush the cache or modify Redis configuration
- Indexing service works normally with restricted command set
- Credential rotation supported

---

#### Issue: Docker Network Segmentation — Single Flat Network for All Services (D-1)

**Problem:** All containers shared a single Docker network. A compromised service could reach any other service directly without network-level isolation.

**Solution:**
- Docker network segmentation planned for future deployment configurations
- Services run with environment variable-based credentials that enforce logical separation
- Network-level restrictions can be implemented at the Docker daemon or orchestrator level

**Impact on Operators:**
- Operators can implement network policies for additional segmentation
- Per-service credentials provide defense-in-depth even without strict network boundaries
- Future releases may add pre-configured network topology

---

#### Issue: Solr Has No Authentication or Authorization Plugins (S-1, S-2)

**Problem:** Solr nodes accepted requests from any client with network access. No user authentication, no per-collection or per-operation authorization.

**Solution:**
- Enabled Solr BasicAuthPlugin for cluster authentication
- Created dedicated users for each service tier:
  - `solr-admin` — Full cluster and collection management (used by `solr-init`)
  - `solr-indexer` — Index write access (used by `document-indexer`)
  - `solr-search` — Read-only search access (used by `solr-search` API)
- Credentials injected via environment variables
- Per-collection ACLs define what each user can do (read, write, admin)
- ZooKeeper-protected ACLs prevent unauthorized config changes

**Impact on Operators:**
- Unauthorized clients cannot read, write, or modify Solr collections
- Service compromise is isolated (indexer cannot delete collections, searcher cannot index)
- Audit trail of who performed what operations (via Solr audit logs)

---

#### Issue: ZooKeeper Has No SASL Authentication or Znode ACLs (Z-1, Z-2)

**Problem:** ZooKeeper ensemble accepted unauthenticated connections and znodes had no ACL restrictions. Any networked client could read or modify cluster configuration.

**Solution:**
- Enabled ZooKeeper SASL authentication (using Digest scheme)
- Created dedicated SASL identities for each service:
  - `solr-admin` — Full access to all znodes (cluster state, configs)
  - `solr-indexer` — Read-only access to cluster state
- Set znode ACLs to restrict access by identity (e.g., only `solr-admin` can modify `/configs`)
- Credentials stored in JAAS config file (injected at container startup)
- Internal ZooKeeper clients (Solr nodes) authenticate via SASL before joining cluster

**Impact on Operators:**
- Unauthorized clients cannot read or modify cluster state
- Configuration changes require authenticated `solr-admin` identity
- Cluster integrity protected from tampering

## Breaking Changes

### Important: Offline Deployment Paths

If you are upgrading from v1.12.x and wish to use the new offline installer:

1. **For online environments:** No action required. Existing deployments continue to work.
2. **For air-gapped environments:** Use the new offline bundle workflow:
   ```bash
   # On networked machine with internet access
   ./scripts/export-images.sh
   
   # Transfer staging/aithena-offline-v*.tar.gz to air-gapped host (via USB, approved media, etc.)
   
   # On air-gapped machine
   tar xzf aithena-offline-v*.tar.gz
   cd aithena-offline-v*/
   sudo ./install.sh
   ```

### Search Behavior Change: Diacritics

- **Default behavior is now diacritic-insensitive.** "Café" matches "cafe".
- Validate search behavior on your indexed data after upgrade to ensure the diacritic-insensitive analyzer works as expected for your library.
- ASCII folding is controlled by the `SOLR_ASCII_FOLDING` environment variable (defaults to true).

### Service Credentials

- **NEW:** RabbitMQ, Solr, ZooKeeper, and Redis now use per-service credentials.
- **Credential Schema Changed:** New per-service environment variables are required. Run the installer or deployment scripts to generate updated `.env` with new credentials.
- Services will not start if credentials are missing or incorrect.

### Network Topology

- All services run on the default Docker network for this release.
- Per-service credentials provide isolation even without network segmentation.
- Future releases may include configurable network policies.

## Security Notes

### Deployment Security Checklist

This release improves security posture through per-service credentials and hardening measures. Before production deployment, verify:

- [ ] **TLS Enabled:** For production, use `docker-compose.ssl.yml` to enable HTTPS
- [ ] **Credentials Unique:** Verify `.env` contains unique, strong credentials for RabbitMQ, Solr, ZooKeeper, and Redis
- [ ] **Non-Root Containers:** Verify `docker compose ps` shows all services running as non-root users
- [ ] **Per-Service Auth:** Verify each service has unique credentials in `.env` (RABBITMQ_INDEXER_USER, RABBITMQ_LISTER_USER, etc.)
- [ ] **Offline Bundle Verified:** If using air-gapped deployment, verify bundle contents and checksums
- [ ] **Firewall Rules:** Restrict direct access to internal ports (Solr 8983, ZooKeeper 2181, Redis 6379, RabbitMQ 5672)
- [ ] **Monitoring Updated:** Point monitoring tools at new service credentials and verify metrics collection

### Security Advisories

- **ASCII Folding:** Enabled by default for improved UX but may have search-ordering implications in some languages. Review search results for your language/script before going live.
- **Offline Bundles:** Treat offline bundles like software releases — sign and verify authenticity before deployment on air-gapped networks.
- **Credential Management:** Store `.env` files with restricted permissions (mode 0600). Do not commit to version control.
- **ZooKeeper AdminServer:** Disabled by default; enable only if cluster management is required, and restrict network access accordingly.

## Upgrade Instructions

### For Online Environments (Recommended)

```bash
# 1. Fetch the new release
git fetch origin
git checkout v1.13.0

# 2. Update Docker images
docker compose pull

# 3. Regenerate .env with new per-service credentials
# (Update any custom settings before restarting)
cp .env.example .env.new
# Edit .env.new with your site-specific settings
# Compare and merge with your current .env

# 4. Restart services with new configuration
docker compose up -d

# 5. Verify all services are healthy
docker compose ps
sleep 30 && docker compose logs | grep -i error
```

### For Air-Gapped Environments

```bash
# 1. On a networked machine with v1.13.0 code:
git fetch origin
git checkout v1.13.0

# 2. Export offline bundle
./scripts/export-images.sh
# Output: staging/aithena-offline-v1.13.0.tar.gz

# 3. Transfer bundle to air-gapped host (via USB, approved media, etc.)

# 4. On the air-gapped host:
tar xzf aithena-offline-v1.13.0.tar.gz
cd aithena-offline-v1.13.0

# 5. Install from offline bundle
sudo ./install.sh

# 6. Verify services are healthy
./verify.sh

# 7. Test search and collections functionality
# (See Validation Checklist below)
```

### Incremental Deployment (Multi-Stage)

If you prefer to stage the security changes gradually:

1. **Stage 1 (Immediate):** Deploy v1.13.0 core and offline installer
2. **Stage 2 (Week 1):** Enable Solr BasicAuth
3. **Stage 3 (Week 2):** Enable ZooKeeper SASL and network segmentation
4. **Stage 4 (Week 3):** Harden RabbitMQ and Redis credentials

Consult the admin manual for per-service hardening procedures.
