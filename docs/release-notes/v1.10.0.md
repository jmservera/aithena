# Aithena v1.10.0 Release Notes — Collections, Metadata Editing & BCDR Foundation

_Date:_ 2026-03-21  
_Prepared by:_ Newt (Product Manager)

## Summary

Aithena v1.10.0 is a major feature release delivering **personal document collections**, **admin metadata editing**, and **disaster recovery infrastructure**. This release enables users to organize and annotate documents, administrators to correct metadata at scale, and operators to protect critical data from loss.

### Key Deliverables

- ✅ **User Collections** — Personal reading lists with notes
- ✅ **Admin Collections Metadata Editing** — Single and batch edit workflows for metadata correction
- ✅ **Folder Hierarchy Filtering** — Hierarchical folder path filtering for precise search
- ✅ **Backup and Disaster Recovery Foundation** — Three-tier backup scripts and restore automation
- ✅ **Stress Testing Framework** — Load testing infrastructure and test data generation
- ✅ **CI/CD Improvements** — Workflow consolidation and security gate enforcement

## Summary of Shipped Changes

### User Collections (Personal Reading Lists)

**New Feature — v1.10.0**

- **Collection Management API** — Full CRUD endpoints (`POST /v1/collections`, `GET /v1/collections`, `PUT /v1/collections/{id}`, `DELETE /v1/collections/{id}`) for creating, listing, editing, and deleting personal collections
- **Collection Items API** — Add/remove documents, bulk operations, note management, reordering within collections
- **Collections UI** — New navigation tab linking to `/collections`, grid view of all collections, detail pages with sortable document lists
- **Search Integration** — "Save to collection" action on search results, collection badges showing which collections contain a document, add-to-collection modal
- **Per-Document Notes** — Users can add notes to any document within a collection, with inline editing and auto-save
- **SQLite Backend** — New `collections.db` at `/data/collections/collections.db` with user ownership enforcement and cascading deletes
- **Search Enrichment** — Search responses now include `in_collections` field showing which user collections contain each document

**Features Implemented:**
- Collection management API (CRUD operations)
- Collections UI with navigation and document lists
- Search integration with "Save to collection" action
- Per-document notes with inline editing
- SQLite backend with user ownership enforcement
- Search response enrichment with collection membership
- Docker volume configuration for persistence
- API endpoints: `POST/GET/PUT/DELETE /v1/collections`, collection items operations

### Metadata Editing (Single & Batch)

**New Feature — v1.10.0**

- **Solr Schema Addition** — New `series_s` field for grouping books into series, magazines, and newspaper collections
- **Single Document Edit API** — `PATCH /v1/admin/documents/{doc_id}/metadata` with atomic Solr updates (title, author, year, category, series)
- **Single Edit UI** — Modal/drawer triggered from search results, pre-filled fields, combobox with existing facet suggestions
- **Batch Edit API** — Endpoints for updating multiple documents by ID list or Solr query with partial-failure reporting
- **Batch Edit UI** — Selection mode with checkboxes, floating action bar, batch editor panel with field toggles and preview
- **Series Facet** — New facet in search sidebar for filtering by series/magazine/newspaper collection
- **Metadata Override Persistence** — Manual edits stored in Redis with re-index safety; overrides survive document requeuing
- **Document-Indexer Integration** — Indexer checks Redis for metadata overrides and applies them during re-indexing

**Features Implemented:**
- Solr atomic updates for metadata changes
- Admin authentication enforcement on all edit endpoints
- Input validation for year, string length, and injection prevention
- Redis override persistence for metadata modifications
- Batch edit API with per-document error reporting
- Series facet integrated into search sidebar
- Single and batch edit APIs: `PATCH /v1/admin/documents/{id}/metadata`, batch edit endpoints

### Backup & Disaster Recovery (BCDR)

**New Feature — v1.10.0 (Foundation)**

- **Critical Tier Backup** (`scripts/backup-critical.sh`) — Auth DB, Collections DB, and secrets backup every 30 minutes with 7-day retention; GPG AES256 encryption, SQLite atomic backups
- **High Tier Backup** (`scripts/backup-high.sh`) — Solr and ZooKeeper snapshots daily (off-peak) with 30-day retention; Solr Collections API BACKUP (non-disruptive)
- **Medium Tier Backup** (`scripts/backup-medium.sh`) — Redis RDB and RabbitMQ definitions daily with 14-day retention
- **Backup Orchestrator** (`scripts/backup.sh`) — Central coordinator with `--tier` selection, `--dry-run` mode, exit code error reporting
- **Restore Orchestrator** (`scripts/restore.sh`) — Per-component restore functions, pre-flight integrity checks, safety backups before overwrite, post-restore verification
- **Backup API Endpoints** — RESTful API for backup status (`GET /v1/admin/backups`), trigger backup (`POST /v1/admin/backups`), restore (`POST /v1/admin/backups/{id}/restore`)
- **Post-Restore Verification** (`tests/verify-restore.sh`) — Automated checklist validating service health, auth, search, Redis, RabbitMQ, Solr cluster status after restore
- **Disaster Recovery Runbook** — `docs/admin/disaster-recovery-runbook.md` with decision tree, 5 recovery paths, step-by-step commands, verification checklists, monthly drill procedure

**Features Implemented:**
- Three-tier backup implementation (critical/30min, high/daily, medium/daily)
- SHA256 checksum verification for backup integrity
- GPG AES256 encryption for sensitive backups
- Graceful file handling for optional components
- Restore orchestrator with integrity checks
- Disaster recovery runbook with recovery paths
- Post-restore verification framework
- Backup API endpoints for management (`GET /v1/admin/backups`, `POST /v1/admin/backups`, `POST /v1/admin/backups/{id}/restore`)

## Security Highlights

### Collections Access Control

- User ownership enforced: users can only view/edit their own collections
- Cross-user access attempts return 403/404
- Cascading deletes respect ownership boundaries

### Metadata Editing Authorization

- All metadata edit endpoints require admin role (`admin_auth.py`)
- Non-admin users receive 403 Forbidden with no data leakage
- Input validation prevents Solr query injection, SQL injection

### Backup System Security

- Auth DB and secrets encrypted with GPG AES256 during backup
- Encryption key stored securely at `/etc/aithena/backup.key` (outside Docker)
- Restore operations checked for backup integrity (SHA256 verification)

### Rate Limiting

- Collections write endpoints rate-limited to 10/minute
- Collection items write endpoints rate-limited to 20/minute
- Bulk operations counted toward items rate limit

## Upgrade Instructions

### For Users Running Aithena v1.9.x

1. **Backup your data:**
   ```bash
   docker compose exec solr-search sqlite3 /data/auth/users.db ".backup /tmp/auth-backup.db"
   ```

2. **Pull the v1.10.0 release:**
   ```bash
   git fetch origin
   git checkout v1.10.0
   ```

3. **Verify schema migrations:**
   Collections DB is created automatically on first startup. Verify with:
   ```bash
   docker compose logs solr-search | grep -i "migration\|collections"
   ```

4. **Restart services:**
   ```bash
   docker compose up -d
   ```

5. **Test user workflows:**
   - Log in and search for a document
   - Click "Save to collection" to test collections feature
   - As admin, click "Edit metadata" on a book to test metadata editing
   - Use folder facet in sidebar to filter by directory

### For Docker Compose Deployments

```bash
docker compose pull
docker compose up -d
```

Schema migrations run automatically. No environment variable changes required.

### Initial Setup for Backup System

To enable automated backups:

1. **Create backup directories:**
   ```bash
   mkdir -p /source/backups/{critical,high,medium}
   chmod 700 /source/backups
   ```

2. **Generate encryption key:**
   ```bash
   openssl rand 32 | base64 > /etc/aithena/backup.key
   chmod 600 /etc/aithena/backup.key
   ```

3. **Install crontab entries:**
   ```bash
   # Every 30 minutes
   */30 * * * *  /path/to/aithena/scripts/backup-critical.sh
   
   # Daily 2 AM UTC
   0 2 * * *     /path/to/aithena/scripts/backup-high.sh
   
   # Daily 3 AM UTC
   0 3 * * *     /path/to/aithena/scripts/backup-medium.sh
   ```

4. **Test first backup manually:**
   ```bash
   ./scripts/backup.sh --tier critical --dry-run
   ./scripts/backup.sh --tier critical
   ```

5. **Verify backup:**
   ```bash
   ls -lah /source/backups/critical/
   ```

See `docs/admin/disaster-recovery-runbook.md` for full setup and recovery procedures.
