# Release Notes — v1.9.0

**Release date:** 2026-03-19  
**Milestone:** [v1.9.0](https://github.com/jmservera/aithena/milestone/23?closed=1)

Aithena v1.9.0 delivers the **User Management & Security** milestone, introducing a complete user management system with full CRUD operations, role-based access control (RBAC), and multiple security hardening measures. This release moves Aithena toward production readiness with configurable password policies, database migrations, and fixes for critical PDF viewer and security vulnerabilities.

---

## Summary of shipped changes

### User Management System

- **User management API** — Complete CRUD endpoints (`POST /v1/auth/register`, `GET /v1/auth/users`, `PUT /v1/auth/users/{id}`, `DELETE /v1/auth/users/{id}`) for admin-driven user lifecycle management (#549, #572)
- **User management UI** — Admin-only `/admin/users` page for viewing, creating, editing, and deleting users; includes user profile page at `/profile` and change password form for self-service (#554, #555, #556, #579, #589)
- **Default admin seeding** — Automatic creation of an initial admin user (`admin`) on first startup if no users exist, simplifying initial deployment (#550)

### Authentication & Security

- **Role-based access control (RBAC)** — Middleware-enforced three-role system:
  - `admin` — Full access to management endpoints, user CRUD, system stats
  - `user` — Standard library search, document upload, profile management
  - `viewer` — Read-only search and document viewing
  - Comprehensive parametrized test suite covering all endpoint × role × auth-state combinations (#553, #559)
- **Password policy enforcement** — Configurable strong password requirements enforced across all code paths:
  - Minimum 10 characters
  - At least 3 of 4 complexity categories (uppercase, lowercase, digits, special characters)
  - Username must not appear in password
  - Policy enforced in API, CLI tools, and admin UI (#552, #574, #584)
- **Change password endpoint** — `PUT /v1/auth/change-password` for authenticated users to update their passwords (#551)
- **Password reset CLI tool** — `reset_password.py` for administrator emergency password recovery, with strong policy validation (#547)
- **Auth database migrations** — Versioned schema migration framework with backup/restore tooling for safe future auth schema changes (#557, #571)
- **Auth integration tests** — Full end-to-end authentication flow validation covering registration, login, role enforcement, and password changes (#558, #575)

### Security Fixes

- **PDF viewer broken** — Fixed X-Frame-Options header preventing iframe embedding; added `X-Frame-Options SAMEORIGIN` to `/documents/` nginx location while preserving `nosniff` protection (#577)
- **Embeddings server outbound requests** — Enforced offline mode with `HF_HUB_OFFLINE=1` and `TRANSFORMERS_OFFLINE=1` environment variables to prevent unintended HuggingFace Hub requests during startup (#578)
- **Login rate limiter ineffective** — Fixed rate limiter reading real client IP from `X-Forwarded-For` header instead of always seeing nginx proxy IP (`172.x.x.x`); prevents IP spoofing and enables proper per-user rate limiting (#583)
- **Password policy not enforced in CLI** — Integrated strong password policy validation into `reset_password.py`; removed misleading 8-character minimum constant from `auth.py` (#584)

### Infrastructure & Documentation

- **Stress testing PRD** — Comprehensive product requirements document for load testing suite and minimum hardware profiling, including goals, test scenarios, metrics framework, and 6-phase implementation roadmap (#590)

## Breaking changes

### Password Policy Upgrade (Required User Action)

- **Minimum password length increased from 8 to 10 characters**
- **Require 3+ complexity categories** (uppercase, lowercase, digits, special characters)
- Existing users with passwords created under the old 8-character policy may be unable to change their password until an administrator resets it to comply with the new policy
- New password changes through the UI enforce the strong policy automatically

### Database Migration

- Automatic schema migrations run on `solr-search` startup
- Safe for production; includes backup/restore tooling
- No data loss; backwards-compatible structure for auth storage

## Security highlights

This release significantly strengthens Aithena's security posture:

- **Authentication isolation:** Three-tier RBAC prevents privilege escalation and limits damage from compromised accounts
- **Password strength:** Enforced 10-character minimum with 3+ complexity categories prevents common weak passwords
- **IP spoofing prevention:** Rate limiter correctly reads client IP from nginx headers, preventing malicious actors from bypassing rate limits
- **Network isolation:** Embeddings server enforces offline mode, preventing exfiltration through HuggingFace Hub
- **Session isolation:** PDF viewer iframe restrictions prevent cross-origin document access
- **CLI tooling:** Admin-only password reset tool enforces same strong policy as web UI

## Upgrade instructions

### For users running Aithena v1.8.x or later

1. **Backup your data:**
   ```bash
   docker compose exec solr-search sqlite3 /data/auth/users.db ".backup /tmp/users-backup.db"
   ```

2. **Pull the v1.9.0 release:**
   ```bash
   git fetch origin
   git checkout v1.9.0
   ```

3. **Verify schema migrations:**
   The auth schema migrations run automatically on first startup. Verify with:
   ```bash
   docker compose logs solr-search | grep -i "migration"
   ```

4. **Reset user passwords to comply with new policy:**
   If you have users with passwords shorter than 10 characters or not meeting the 3+ complexity category requirement, reset their passwords using the CLI tool:
   ```bash
   docker compose exec solr-search python reset_password.py --username <username>
   ```
   The tool generates a secure 32-character random password if `--password` is omitted.

5. **Verify role-based access control:**
   - Log in as `admin` — confirm access to `/admin/users` page
   - Verify `user` role can access search and upload
   - Verify `viewer` role can only access search results (no upload/admin features)

6. **Test password reset flow:**
   - Use the change password form at `/profile` to update your password
   - Confirm the new password meets the 10-character, 3+ complexity requirement

### For Docker Compose deployments

```bash
docker compose pull
docker compose up -d
```

The migrations run automatically. Monitor logs with:
```bash
docker compose logs -f solr-search
```

### Environment variables (unchanged)

No new environment variables required. All defaults are production-safe:
- `RBAC_DEFAULT_ROLE=user` — new users default to `user` role (not `admin` or `viewer`)
- Password policy is built-in; no configuration needed

### Rate limiting updates

The login rate limiter now correctly enforces per-IP limits thanks to the `X-Forwarded-For` fix. If you have custom nginx proxy configurations, verify they set:
```nginx
proxy_set_header X-Forwarded-For $remote_addr;
```

## Validation checklist

Before using v1.9.0 in production, verify:

- [ ] Application starts and migrations complete: `docker compose up -d && docker compose logs solr-search | grep -i migration`
- [ ] Default admin user exists and can log in
- [ ] Admin user can navigate to `/admin/users` and see empty user list
- [ ] Admin user can create a new user with a strong password (10+ chars, 3+ complexity)
- [ ] New user can log in and access search/upload but not admin features
- [ ] Admin can reset another user's password via CLI tool
- [ ] Login rate limiting works: send 6 login requests in 60 seconds and confirm 429 response on 6th attempt
- [ ] PDF viewer works: open a document from search results
- [ ] Embeddings server enforces offline mode: `docker compose logs embeddings-server | grep -i offline`
- [ ] No console errors in browser DevTools
- [ ] All services start cleanly: `docker compose up -d`
