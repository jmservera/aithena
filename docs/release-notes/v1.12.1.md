# Aithena v1.12.1 Release Notes — Collections API Fix & UX Refinements

_Date:_ 2026-03-23  
_Prepared by:_ Newt (Product Manager)

## Summary

Aithena v1.12.1 is a maintenance release focusing on **bug fixes and UX refinements** introduced by v1.11.0. The release fixes the collections feature to use real backend API data by default, adds the missing "Remember me" checkbox to the login form, truncates lengthy text previews in search results, and resolves a container build issue preventing thumbnail generation in Alpine-based environments. The release closed 4 critical issues via bug fixes and enhancements.

### Key Deliverables

- ✅ **Collections API Real Data** — Fixed mock data fallback; collections now use real backend API by default
- ✅ **Remember Me Checkbox** — Added session persistence control to login form
- ✅ **Text Preview Truncation** — Search result chunk text now truncated to ~15-20 characters with context
- ✅ **Container Build Fix** — Added missing `libstdc++` to Alpine Linux runtime; thumbnail generation now works

## Shipped Changes

### Issue #897: Enable Real Collections API

**Problem:** Collections feature was fully implemented on backend and frontend, but the frontend API service (`collectionsApi.ts`) defaulted to mock data instead of calling the real backend API.

**Root Cause:** `src/aithena-ui/src/services/collectionsApi.ts` checked for a `VITE_COLLECTIONS_API === "real"` environment variable before making actual API calls. Without this variable, all collection operations returned hardcoded mock data, making the feature non-functional in production.

**Fix:**
- Removed the mock data fallback from the collections API service
- Collections API now uses real backend endpoints by default
- No environment variable configuration required

**Impact on Users:**
- Users can now create, edit, and delete collections with persistent data storage
- Added books are stored in the backend SQLite database
- Collection notes and book reordering are preserved across sessions

**Verification:**
- [ ] Create a new collection from the UI
- [ ] Add books from search results to the collection
- [ ] Add notes to collection items
- [ ] Reload the page and verify collection data persists
- [ ] Delete a collection and verify it is removed

### Issue #898: Add "Remember Me" Checkbox to Login Form

**Problem:** The backend fully supported the "Remember me" feature (JWT `rmb` claim, persistent cookie configuration) but the frontend never sent the `remember_me` flag in login requests. The checkbox was missing from the login form with an explicit TODO comment.

**Root Cause:** `LoginPage.tsx` lacked a UI checkbox and `AuthContext.tsx` login function did not accept a `rememberMe` parameter.

**Fix:**
- Added "Remember me" checkbox input to `LoginPage.tsx`
- Updated `AuthContext.login()` signature to accept optional `rememberMe` parameter
- Login requests now include `remember_me` in the POST body
- Session behavior now respects the user's choice:
  - **Checked:** Session persists for 30 days (persistent cookie)
  - **Unchecked:** Session ends when browser is closed (session cookie)

**Impact on Users:**
- Users can now opt for persistent login sessions by checking "Remember me"
- Default behavior (unchecked) maintains secure session-only authentication
- Users who check "Remember me" will remain logged in across browser restarts

**Verification:**
- [ ] Checkbox visible and labeled on login form
- [ ] Checking "Remember me" creates a persistent session cookie
- [ ] Unchecking (default) creates a session-only cookie that expires on browser close
- [ ] Logout clears the session regardless of remember-me setting

### Issue #896: Truncate Text Preview in Search Results

**Problem:** Chunk text previews in search results (BookCard component) displayed in full, sometimes spanning 200+ characters and cluttering the UI. The text preview field appeared disproportionately large on result cards.

**Root Cause:** `BookCard.tsx` rendered `book.chunk_text` without any character or line-clamp limits, even though Solr highlights were already truncated to 160 characters.

**Fix:**
- Implemented truncation logic in `BookCard.tsx` to limit chunk text to approximately 15-20 characters
- For keyword/hybrid searches, truncation centers around the highlighted `<em>` term to keep the search result visible
- For semantic search, chunk text is truncated with ellipsis
- BookDetailView continues to show full chunk text (no truncation)

**Impact on Users:**
- Search result cards are more compact and readable
- Highlighted search terms remain visible and centered in the truncated preview
- Users can still view full text by opening the book detail view
- Search results are less visually overwhelming

**Verification:**
- [ ] Chunk text in BookCard truncated to ~15-20 characters
- [ ] Highlighted term remains visible in truncated text
- [ ] BookDetailView still shows full chunk text (no truncation)
- [ ] Truncation works for both keyword and semantic searches

### Issue #894: Fix Thumbnail Generation — Missing libstdc++ in Alpine Container

**Problem:** Thumbnail generation in `document-indexer` failed at container runtime with:
```
ImportError: Error loading shared library libstdc++.so.6: No such file or directory
(needed by /app/.venv/lib/python3.12/site-packages/pymupdf/_extra.so)
```

Documents still indexed (thumbnail generation is best-effort), but no thumbnails were generated for any PDFs.

**Root Cause:** The Alpine Linux base image (`python:3.12-alpine`) used for `document-indexer` does not include the C++ standard library (`libstdc++.so.6`), which is a runtime dependency of the PyMuPDF compiled extensions. The build stage successfully downloaded the precompiled wheel, but the runtime stage was missing the required shared library.

**Fix:**
- Updated `src/document-indexer/Dockerfile` to include missing C++ runtime libraries
- Added `apk add --no-cache libstdc++ libgomp libgcc` to the runtime stage
- Thumbnail generation now succeeds at container startup

**Impact on Users:**
- Thumbnails are now generated during document indexing
- Thumbnail images appear in search results and library views
- PDF first-page previews load correctly

**Impact on Operators:**
- Containers now include required C++ libraries for PyMuPDF
- No additional host configuration needed
- Existing documents indexed without thumbnails can be reindexed to generate missing thumbnails

**Verification:**
- [ ] document-indexer container starts without libstdc++ errors
- [ ] Container logs show no PyMuPDF import errors
- [ ] New PDFs indexed have thumbnails generated
- [ ] Thumbnails load in search results and library views

## Upgrade Instructions

### Standard Upgrade (Recommended)

```bash
git fetch origin
git checkout v1.12.1
docker compose pull
docker compose up -d
```

### With Thumbnail Reindexing

If you deployed v1.11.0 with thumbnail generation failing, reindex documents after upgrading:

```bash
# 1. Deploy v1.12.1
git fetch origin
git checkout v1.12.1
docker compose pull
docker compose up -d

# 2. Wait for services to stabilize
sleep 60 && docker compose ps

# 3. Clear existing index (if desired) and trigger reindex
# Consult your indexing pipeline configuration for specific commands

# 4. Monitor reindexing progress
docker compose logs -f document-indexer
```
