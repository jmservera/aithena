## Shipped Changes

### Feature Track: Embedding Model Evaluation

#### Issue #926: Evaluate e5-base vs distiluse embeddings — benchmark and quality comparison

**Problem:** Aithena v1.12.0 introduced dual embedding infrastructure capable of running both `distiluse-base-multilingual-cased-v2` (current default) and `multilingual-e5-base` (candidate replacement) in parallel. However, without a systematic evaluation framework, teams could not objectively compare model performance and quality.

**Solution:**
- A/B evaluation infrastructure from v1.12.0 is now fully instrumented
- Benchmark API endpoint (`/v1/search/compare`) enables parallel queries against both models
- Test corpus indexing workflow documented and automated for reproducible evaluation
- 30-query benchmark suite provided covering:
  - **Exact Match** — Term searches that should prioritize document frequency
  - **Semantic** — Conceptual queries requiring embedding understanding
  - **Multilingual** — Queries in multiple languages to test cross-lingual capabilities
  - **Fuzzy** — Misspellings and partial terms
  - **Complex** — Combined fuzzy + semantic + multiword queries
- Metrics collection framework: latency (p50, p95, p99), memory usage, index size
- Quality assessment: relevance scoring, nDCG@10, manual spot-check procedures

**Evaluation Decision Gate:**
- **If performance loss is negligible** → migrate directly to e5-base, skip human evaluation UI (v1.14.0)
- **If significant quality differences** → proceed with v1.14.0 evaluation UI for human raters

**User Impact:**
- No user-facing changes in v1.13.1 — embedding model remains `distiluse-base-multilingual-cased-v2`
- Evaluation results inform future releases (v1.14.0 or direct migration to e5-base)
- Teams can now run objective performance benchmarks internally

**Evaluation Checklist:**
- [ ] Build and start stack: `./buildall.sh && docker compose up -d`
- [ ] Index test corpus into both `books` (distiluse) and `books_e5base` (e5-base)
- [ ] Verify document counts match across collections
- [ ] Run 30-query benchmark through comparison API (`/v1/search/compare`)
- [ ] Collect latency metrics: p50, p95, p99 per collection
- [ ] Collect memory usage per embeddings server
- [ ] Compare index size between models
- [ ] Perform relevance assessment: nDCG@10 if ground truth available
- [ ] Manual spot-check for multilingual and semantic query quality
- [ ] Document recommendation: migrate or proceed to v1.14.0 evaluation
