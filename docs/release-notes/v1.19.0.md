## Summary

Aithena v1.19.0 is a **minor release** focused on infrastructure cleanliness and operational flexibility:

1. **Deprecation warnings suppressed (#1432)** — Eliminates noisy Solr 9.7 Security Manager and RabbitMQ 4.x deprecation warnings from container logs
2. **Configurable Solr topology (#1428)** — `SOLR_NUM_SHARDS` and `SOLR_REPLICATION_FACTOR` environment variables allow operators to tune collection topology at init time
3. **Preflight validation for topology variables (#1440)** — `solr-init` validates shard and replication factor values before attempting collection creation
4. **38+ dependency updates** — Security and compatibility patches across all services
5. **CI/CD improvements (#1413, #1415)** — Dependabot batch merge workflow and automerge hardening

There are **no user-facing changes** and **no breaking changes**. Upgrade is a standard `docker compose pull && docker compose up -d`.

## Verified Scope for v1.19.0

### Fix 1: Deprecation Warnings Suppressed (#1432, PR #1432)

**Problem:**  
Eleven deprecation warnings were detected during pre-release testing (#1367):
- **Solr Security Manager warnings** — Solr nodes `solr`, `solr2`, `solr3` each emitted a Java Security Manager deprecation warning on every startup
- **RabbitMQ deprecated_features warning** — `rabbitmq.conf` contained `deprecated_features.permit.management_metrics_collection = false`, which triggered a deprecation warning on RabbitMQ 4.x startup

**Root Cause:**  
- The `solr-init` service already had `SOLR_SECURITY_MANAGER_ENABLED=false`, but the three runtime Solr nodes did not, causing them to emit the warning
- The `rabbitmq.conf` setting references an in-memory metrics collection feature that RabbitMQ 4.x disables by default; explicitly referencing it triggered a deprecation notice

**Fix:**
1. Added `SOLR_SECURITY_MANAGER_ENABLED=false` to the `solr`, `solr2`, and `solr3` service definitions in the Docker Compose files
2. Removed `deprecated_features.permit.management_metrics_collection = false` from `rabbitmq.conf`

**Files Changed:**
- `docker-compose.yml` — added env var to solr, solr2, solr3 services
- `docker-compose.prod.yml` — same
- `rabbitmq.conf` (or equivalent RabbitMQ config file) — removed deprecated setting

**Tests:** No new tests required — this is a configuration-only change. Log output is the observable signal.

**Impact:** Container logs are significantly cleaner after upgrade. No functional behavior changes.

**Not addressed (upstream):**
- Solr `-type` CLI warning: the codebase already uses `--type` (double-dash); this is an internal Solr 9.7 CLI parsing message
- Jetty deprecation warnings: upstream Solr/Jetty issue, not configurable

### Feature 1: Configurable Solr Topology (#1428)

**Problem:**  
The `books` collection was always created with 1 shard and replication factor 3 (one replica per node), which is unnecessarily heavy for personal or single-node deployments and inflexible for large libraries requiring more shards.

**Fix:**  
Added `SOLR_NUM_SHARDS` and `SOLR_REPLICATION_FACTOR` environment variables to `solr-init`. Defaults retain existing behavior (1 shard, 3 RF).

**Files Changed:**
- `docker-compose.yml` — env var declarations
- `src/solr-search/` — solr-init entrypoint updated to consume the new variables

**Impact:**
- Personal deployments can set `SOLR_REPLICATION_FACTOR=1` to save ~⅔ of disk and memory
- Large libraries (30K+ docs) can increase shards for query performance
- Existing deployments are unaffected (defaults unchanged)

> ⚠️ These variables only take effect when the collection is first created. Changing them on an existing deployment requires deleting and recreating the collection, which triggers a full re-index.

---

### Feature 2: Preflight Topology Validation (#1440)

**Problem:**  
Invalid values for `SOLR_NUM_SHARDS` or `SOLR_REPLICATION_FACTOR` would only fail at collection creation time, producing an unclear Solr error deep in init logs.

**Fix:**  
`solr-init` now validates both variables before attempting to create the collection, emitting a clear error message and exiting early if values are invalid.

**Impact:** Operators receive actionable error messages immediately on misconfiguration rather than cryptic Solr API errors.

## Deployment Notes

### Upgrade Path

**From v1.18.1 → v1.19.0:**

1. **Pull new images:**
   ```bash
   docker compose pull
   ```

2. **(Optional) Configure Solr topology** in `.env` if you want non-default values:
   ```bash
   SOLR_NUM_SHARDS=1
   SOLR_REPLICATION_FACTOR=1   # for single-node/personal deployments
   ```

3. **Restart all services:**
   ```bash
   docker compose up -d
   ```

4. **Verify clean startup logs:**
   ```bash
   docker compose logs --tail=20 solr
   docker compose logs --tail=20 rabbitmq
   ```
   You should no longer see Security Manager or `deprecated_features` warnings.

**From v1.17.x or earlier:**  
Follow the [v1.18.0 deployment notes](v1.18.0.md#deployment-notes) and [v1.18.1 deployment notes](v1.18.1.md#deployment-notes) first, then apply the v1.19.0 steps above.

### Configuration Changes

| Change | File | Required action |
|---|---|---|
| `SOLR_NUM_SHARDS` / `SOLR_REPLICATION_FACTOR` | `.env` | **Optional.** Only set if you want non-default topology. |
| `SOLR_SECURITY_MANAGER_ENABLED=false` | `docker-compose.yml` | **Automatic.** Applied via updated compose file. |
| Removed `deprecated_features` setting | `rabbitmq.conf` | **Automatic.** Applied via updated config file. |

### Data Migration

**None required.** The Solr collection schema, index format, and auth database are unchanged. Existing data is fully compatible.
