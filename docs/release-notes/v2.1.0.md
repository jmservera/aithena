# Aithena v2.1.0 Release Notes — Configurable Search Architecture and Single-Node Deployment

_Date:_ 2026-04-21
_Prepared by:_ Copilot

## Summary

Aithena v2.1.0 is a **minor release** focused on deployment flexibility and resource optimization. The headline feature is configurable search architecture — operators can now choose between full HNSW vector search (current behavior) or a hybrid-rerank mode that eliminates the HNSW index entirely, reducing RAM requirements by up to 28GB for large deployments. This release also introduces single-node Docker Compose topology, single-user mode for simplified local development, configurable vector quantization, and comprehensive CI/CD improvements for faster validation cycles.

### Highlights

1. **Configurable search architecture (HNSW vs hybrid-rerank)** — New `SEARCH_ARCHITECTURE=hnsw|hybrid-rerank` environment variable lets operators choose between full kNN vector search (9-28GB HNSW RAM) and a hybrid-rerank mode that uses BM25 pre-filtering + numpy cosine reranking (zero HNSW RAM). The UI dynamically adapts based on backend capabilities.
2. **Single-node Docker Compose topology** — New `docker/compose.single-node.yml` overlay replaces 3 ZK + 3 Solr (~12GB) with 1 ZK + 1 Solr (~4GB) while maintaining full functionality including RBAC and auth. Ideal for personal use, development, and <50K document deployments.
3. **Configurable vector quantization** — New `VECTOR_QUANTIZATION=none|fp16|int8` environment variable reduces HNSW memory from ~28GB (float32) to ~14GB (fp16) or ~9GB (int8) for 9M vectors with minimal search quality impact (within 1-3%).
4. **Single-user mode (`AUTH_ENABLED=false`)** — Simplifies local development and CI environments by disabling authentication entirely (no auth DB, no user/password prompts, no JWT secret, no ADMIN_API_KEY).
5. **Installer topology selection** — New `--topology single-node|distributed` flag streamlines initial deployment with sensible defaults (single-node for new installs, distributed for production).
6. **CI/CD improvements** — New dev integration test workflow targets <20 minutes (vs ~75 min distributed) using single-node topology. Release CI now validates both topologies and both auth modes with a clear matrix strategy.
7. **Deployment topologies documentation** — New `docs/deployment-topologies.md` explains resource requirements, use cases, and migration paths for both topologies.

---

## Breaking Changes

**None for default configuration.** All new features default to current behavior:
- `SEARCH_ARCHITECTURE` defaults to `hnsw` (existing HNSW-based search)
- `VECTOR_QUANTIZATION` defaults to `none` (float32, current behavior)
- `AUTH_ENABLED` defaults to `true` (multi-user authentication enabled)
- Default installer topology is `single-node` for **new installs**, but existing deployments retain their configuration

Operators who want to switch to hybrid-rerank mode, enable quantization, or change topologies should review the upgrade instructions below.

## Upgrade Path

### From v2.0.0 → v2.1.0

1. **Back up your data** (recommended for production):
   ```bash
   # Export Solr collections
   docker compose exec solr bin/solr export -collection books -out /var/solr/data/backup
   ```

2. **Pull new images:**
   ```bash
   docker compose pull
   ```

3. **Review new environment variables** (optional — defaults preserve v2.0.0 behavior):
   - `SEARCH_ARCHITECTURE=hnsw` (default) — Keep current HNSW behavior
   - `VECTOR_QUANTIZATION=none` (default) — Keep float32 vectors
   - `AUTH_ENABLED=true` (default) — Keep authentication enabled

4. **Restart services:**
   ```bash
   docker compose up -d
   ```

5. **Verify health:**
   ```bash
   curl -sf http://localhost/v1/health && echo "✅ API healthy"
   curl -sf http://localhost/v1/capabilities && echo "✅ Capabilities endpoint responding"
   ```

### Switching to Hybrid-Rerank Mode (Optional)

**Use case:** Reduce HNSW vector RAM from 9-28GB to zero at 30K book scale. Suitable for memory-constrained deployments.

**Trade-offs:**
- "Semantic" search mode is hidden from UI (only Keyword and Hybrid remain)
- Similar Books feature is disabled
- Search quality relies on BM25 pre-filtering + stored vector reranking (no HNSW graph)

**Steps:**

1. **Update `.env`:**
   ```bash
   SEARCH_ARCHITECTURE=hybrid-rerank
   ```

2. **Restart services:**
   ```bash
   docker compose down
   docker compose up -d
   ```

3. **Reindex your library:**
   ```bash
   # Trigger reindex via admin portal or API
   curl -X POST http://localhost/v1/admin/reindex
   ```

   > **Note:** Switching from HNSW to hybrid-rerank (or vice versa) requires reindexing because the HNSW graph must be built or discarded.

4. **Verify capabilities:**
   ```bash
   curl -s http://localhost/v1/capabilities | jq .
   # Should show: {"architecture": "hybrid-rerank", "search_modes": ["keyword", "hybrid"], ...}
   ```

### Switching to Single-Node Topology (Optional)

**Use case:** Reduce infrastructure footprint from ~12GB (3 ZK + 3 Solr) to ~4GB (1 ZK + 1 Solr). Suitable for personal use, development, or <50K documents.

**Steps:**

1. **Update your docker compose command:**
   ```bash
   docker compose -f docker-compose.yml -f docker/compose.single-node.yml up -d
   ```

   Or re-run the installer with `--topology single-node` to regenerate `start.sh`.

2. **Existing data is preserved.** Solr will replicate data to the new single-node cluster.

See [Deployment Topologies Documentation](../deployment-topologies.md) for detailed migration instructions.

## What's New in Detail

### Configurable Search Architecture (HNSW vs Hybrid-Rerank)

**Issue:** #1506

Aithena now supports two search architectures controlled by the `SEARCH_ARCHITECTURE` environment variable:

#### HNSW Mode (Default)

- **Current behavior:** Full kNN vector search with HNSW index
- **All three search modes available:** Keyword, Semantic, Hybrid
- **HNSW RAM requirement:** 9-28GB for page vectors (depends on library size)
- **Similar Books:** Enabled (uses HNSW kNN)
- **Best for:** Production deployments with >50K documents where search quality is the top priority

#### Hybrid-Rerank Mode (New)

- **Zero HNSW RAM:** No HNSW index is built or loaded
- **Search flow:**
  1. BM25 finds top-200 keyword candidates
  2. Stored vectors retrieved from Solr
  3. Numpy cosine similarity reranking
  4. RRF fusion combines keyword and semantic scores
- **Two search modes available:** Keyword, Hybrid (Semantic is hidden from UI)
- **Similar Books:** Disabled (requires HNSW index)
- **Best for:** Memory-constrained deployments (<32GB RAM) with <50K documents where infrastructure cost is a concern

#### Capabilities Endpoint

New `/v1/capabilities` endpoint returns backend configuration:

```json
{
  "search_modes": ["keyword", "semantic", "hybrid"],
  "architecture": "hnsw",
  "vector_quantization": "none",
  "vector_dimensions": 768
}
```

The UI uses this endpoint to dynamically show/hide search modes based on backend capabilities.

#### Switching Architectures

**Important:** Switching architectures requires reindexing because the HNSW graph must be built (for `hnsw`) or discarded (for `hybrid-rerank`).

**Motivation:** At 30K book scale with 9M page vectors, the HNSW index requires 9-28GB RAM (depending on quantization). For cost-constrained deployments, hybrid-rerank eliminates this requirement entirely while maintaining good search quality for most use cases.

### Configurable Vector Quantization

**Issue:** #1502

New `VECTOR_QUANTIZATION` environment variable controls how vectors are encoded in the HNSW index:

| Mode | Encoding | HNSW RAM (9M vectors) | Search Quality Loss |
|------|----------|----------------------|---------------------|
| `none` (default) | float32 | ~28GB | 0% (baseline) |
| `fp16` | float16 | ~14GB | <1% |
| `int8` | BYTE | ~9GB | <3% |

**Implementation:**
- Embeddings server validates quantization quality at startup (logs warning if cosine similarity degrades beyond threshold)
- Solr schema includes new `knn_vector_768_byte` field type for `int8` mode with `vectorEncoding="BYTE"`
- Switching quantization modes requires reindexing (HNSW graph uses different field)

**Use cases:**
- `fp16`: Recommended for production — 50% RAM savings with negligible quality loss
- `int8`: Best for cost-constrained deployments — 68% RAM savings with acceptable quality loss (<3%)
- `none`: Baseline — no quantization, maximum search quality

### Single-Node Docker Compose Topology

**Issue:** #1495

New `docker/compose.single-node.yml` overlay replaces the distributed Solr cluster with a single-node configuration:

**Distributed (Default in v2.0.0):**
- 3 ZooKeeper nodes
- 3 Solr nodes
- ~12GB RAM total
- High availability, automatic failover
- Best for production (>50K documents)

**Single-Node (New in v2.1.0):**
- 1 ZooKeeper node
- 1 Solr node
- ~4GB RAM total
- No high availability
- Best for personal use, development, <50K documents

**All services unchanged** except Solr cluster. RBAC, authentication, and all features remain fully functional.

**Usage:**
```bash
docker compose -f docker-compose.yml -f docker/compose.single-node.yml up -d
```

See [Deployment Topologies Documentation](../deployment-topologies.md) for detailed resource requirements and migration instructions.

### Single-User Mode (`AUTH_ENABLED=false`)

**Issues:** #1490, #1493

New `AUTH_ENABLED` environment variable simplifies local development and CI environments:

**Multi-user mode (`AUTH_ENABLED=true`, default):**
- Auth database required
- User/password prompts at first login
- JWT secret required
- ADMIN_API_KEY required for admin endpoints

**Single-user mode (`AUTH_ENABLED=false`):**
- No auth database
- No user/password prompts
- No JWT secret
- No ADMIN_API_KEY
- All API endpoints publicly accessible

**Installer support:**
- New `--no-auth` / `--single-user` flag skips auth setup and sets `AUTH_ENABLED=false` in `.env`

**Use cases:**
- Local development (no authentication overhead)
- CI/CD environments (simplified test setup)
- Single-user personal deployments behind a firewall

**Security warning:** Single-user mode disables all authentication and authorization. Only use in trusted environments.

### Installer Topology Selection

**Issue:** #1497

The installer now prompts for deployment topology:

**New `--topology` flag:**
- `--topology single-node` — Sets `SOLR_NUM_SHARDS=1`, `SOLR_REPLICATION_FACTOR=1` in `.env`
- `--topology distributed` — Configurable shards/replicas (default: 1 shard, 3 replicas)

**Default:** `single-node` for new installs (optimized for getting started quickly with minimal resources)

**Generated `.env` includes compose command hint:**
```bash
# For single-node topology:
# docker compose -f docker-compose.yml -f docker/compose.single-node.yml up -d

# For distributed topology:
# docker compose up -d
```

**Backward compatible:** Existing `.env` files continue to work without modification.

## Known Issues

### 1. Switching Search Architecture Requires Reindexing

Changing `SEARCH_ARCHITECTURE` between `hnsw` and `hybrid-rerank` requires a full reindex because the HNSW graph must be built or discarded. Plan for indexing downtime when switching architectures.

### 2. Hybrid-Rerank Mode Limitations

In hybrid-rerank mode:
- Semantic search mode is hidden (only Keyword and Hybrid available)
- Similar Books feature is disabled (requires HNSW index)
- Search quality depends on BM25 pre-filtering effectiveness (may be lower than HNSW for certain queries)

Operators should evaluate search quality in hybrid-rerank mode on a staging environment before deploying to production.

### 3. Vector Quantization Quality Threshold

The embeddings server validates quantization quality at startup. If cosine similarity between original and quantized vectors degrades beyond a threshold, a warning is logged. This warning is informational only — the service continues to run. Operators should monitor search quality in production.
