## Verified Scope for v1.18.1

### Fix 1: Solr Auth Role Assignment (#1332, PR #1333)

**Problem:**  
Solr auth bootstrap was overwriting admin user roles, stripping critical permissions needed for security operations (`superadmin`), collection admin read (`search`), and document updates (`index`).

**Root Cause:**  
Solr 9.7's `solr auth enable` automatically assigns all 4 built-in roles (`superadmin`, `admin`, `search`, `index`) to the admin user. The `solr-init` entrypoint was subsequently calling `set-user-role` to overwrite with only `["admin"]`, inadvertently removing the other three roles.

**Fix:**
1. Removed the redundant `set-user-role` call for the admin user — `solr auth enable` already handles role assignment correctly
2. Changed readonly user role from custom `"readonly"` to Solr 9.7 built-in `"search"`
3. Updated `security.json` to match Solr 9.7's 4-tier role hierarchy:
   - `superadmin` — security-edit permissions
   - `admin` — security-read, config-edit, core/collection admin
   - `search` — collection-admin-read, read
   - `index` — update
4. Updated tests to verify new behavior

**Files Changed:**
- `docker-compose.yml`
- `docker-compose.prod.yml`
- `src/solr/security.json`
- `src/solr-search/tests/test_solr_init_script.py`

**Tests:** Updated test suite passes, verifying role assignments align with Solr 9.7 defaults.

**Impact:** Admin users now retain full permissions for security operations, collection management, and document indexing.

---

### Fix 2: Installer CWD Independence (#1330, PR #1331)

**Problem:**  
Running `uv run installer/setup.py` from the repository root failed with `ModuleNotFoundError: No module named 'aithena_common'` because `uv` resolves dependencies from the current working directory's `pyproject.toml`, and the repo root lacks one.

**Root Cause:**  
The installer script depends on `aithena-common`, but without a project file in the repo root, `uv` couldn't resolve the dependency when invoked from outside the `installer/` directory.

**Fix:**  
Added PEP 723 inline script metadata to `installer/setup.py`:
```python
# /// script
# dependencies = ["aithena-common"]
# ///
```

This tells `uv run` about the `aithena-common` dependency regardless of CWD. When run from `installer/`, existing behavior is unchanged. When run from the repo root, the inline metadata resolves the path relative to the script location.

**Files Changed:**
- `installer/setup.py`

**Tests:**  
All 6 installer tests pass:
- ✅ `cd /repo && uv run installer/setup.py --help` (from repo root)
- ✅ `cd /repo/installer && uv run setup.py --help` (from installer dir)

**Impact:** Users can now run the installer from any directory without encountering import errors.

## Deployment Notes

### Upgrade Path

**From v1.18.0 → v1.18.1:**

1. **Pull the new image:**
   ```bash
   docker compose pull
   ```

2. **Recreate Solr container** (to apply auth role fix):
   ```bash
   docker compose up -d --force-recreate solr
   ```

3. **Verify Solr admin roles:**
   ```bash
   docker compose exec solr curl -u admin:admin \
     "http://localhost:8983/solr/admin/authentication" | jq '.authentication.credentials'
   ```
   
   The admin user should have roles: `["superadmin", "admin", "search", "index"]`

4. **Verify installer works from repo root** (optional, for maintainers):
   ```bash
   uv run installer/setup.py --help
   ```

**From v1.17.x or earlier:**  
Follow the [v1.18.0 deployment notes](v1.18.0.md#deployment-notes) first, then apply v1.18.1 updates above.

### Configuration Changes

**None required.** The fixes are applied automatically via updated container images and script metadata.

### Data Migration

**None required.** Solr's existing security configuration is updated in-place on container restart.
