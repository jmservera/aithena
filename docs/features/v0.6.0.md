# v0.6.0 Feature Guide

Aithena v0.6.0 introduces **PDF upload capability** for users, **security scanning infrastructure** for continuous vulnerability detection, and **production-grade container hardening** with health checks and resource limits. This guide describes the shipped behavior verified in the current source code.

## Feature summary

- **PDF Upload Flow** with drag-and-drop UI, client-side validation, and streaming backend processing
- **Backend Upload Endpoint** (POST /v1/upload) with MIME/extension/magic-byte validation and rate limiting
- **Security Scanning in CI** with bandit (Python), checkov (IaC), and zizmor (GitHub Actions)
- **Security Baseline Document** enumerating 287 findings triaged to guide v0.6 and v0.7 roadmap
- **Docker Health Checks** for all services with proper startup sequencing and failure recovery
- **Resource Limits** on memory and CPU for production stability
- **Restart Policies** ensuring graceful recovery from transient failures

## PDF Upload Feature

### User-facing Upload UI

The frontend exposes a dedicated **Upload** tab in the main navigation bar.

#### What users see

- **Drag-and-drop zone** — users can drag PDFs directly onto the page
- **File picker** — button to browse and select files alternatively
- **Progress bar** — real-time upload progress as the file transfers
- **Error feedback** — inline error messages for rejected files
- **Success confirmation** — visual indication when upload completes

#### Client-side validation

Before sending to the server, the UI validates:

- **File type**: Must be a PDF (`application/pdf`)
- **File size**: Maximum 50 MB
- **Error states**: If validation fails, the UI shows:
  - "Invalid file type. Please upload a PDF."
  - "File is too large. Maximum size is 50 MB."

#### Upload progress tracking

- Uses `XMLHttpRequest` for fine-grained progress tracking
- Displays bytes uploaded vs. total file size
- Shows upload speed and estimated time remaining
- User can view the upload without navigating away

#### Error handling

The UI gracefully handles all backend HTTP response codes:

- **400 Bad Request** → "Invalid file. Server rejected this upload."
- **413 Payload Too Large** → "File is too large. Maximum size is 50 MB."
- **429 Too Many Requests** → "Too many uploads. Please wait a moment and try again."
- **500 Internal Server Error** → "Upload failed. Please try again."
- **502 Bad Gateway** → "Service temporarily unavailable. Please try again later."

### Backend Upload Endpoint

#### POST /v1/upload

Accepts multipart form data with a single PDF file.

**Request format:**
```
POST /v1/upload HTTP/1.1
Content-Type: multipart/form-data

--boundary
Content-Disposition: form-data; name="file"; filename="example.pdf"
Content-Type: application/pdf

[binary PDF content]
--boundary--
```

**Response (201 Created):**
```json
{
  "status": "accepted",
  "filename": "example.pdf",
  "message": "File queued for indexing"
}
```

#### Validation pipeline

The backend performs **three layers** of validation:

1. **MIME type check** — rejects if `Content-Type` is not `application/pdf`
2. **Filename extension check** — rejects if filename does not end in `.pdf`
3. **PDF magic bytes check** — rejects if file does not start with `%PDF`

If any layer fails, the endpoint returns `400 Bad Request` with a descriptive error message.

#### Streaming upload processing

- Files are read in **8 KB chunks** to prevent out-of-memory errors on large files
- If validation fails, the stream is aborted immediately without consuming the full payload
- Partial uploads are not retained

#### Rate limiting

- **Limit**: 10 uploads per minute per client IP
- **Strategy**: Sliding window counter stored in Redis
- **Response**: HTTP 429 Too Many Requests when limit exceeded
- **Retry-After header**: Included in 429 response to guide clients

#### Path traversal prevention

- Filenames are sanitized to reject path traversal sequences (`../`, `..\\`, absolute paths)
- Only the base filename is used; directory components are stripped
- Uploaded files are placed in a temporary staging directory with randomized names

#### Integration with indexing pipeline

- Upon success, the uploaded file is placed in `/data/documents/uploads/` within the container
- An entry is queued to the `shortembeddings` RabbitMQ queue
- The document-indexer processes the file just like any discovered document
- Metadata is extracted from the filename if it follows known patterns; otherwise defaults are used

## Security Scanning

### Bandit Python Security Scanning (SEC-1)

#### What it does

Bandit scans Python source code for common security issues:
- SQL injection vulnerabilities
- Hardcoded secrets (passwords, tokens)
- Insecure random number generation
- Use of unsafe deserialization
- Weak cryptographic algorithms

#### How it runs

- **Trigger**: Every push to any branch
- **Workflow**: `.github/workflows/security-bandit.yml`
- **Scope**: All Python files in `solr-search/`, `document-indexer/`, `document-lister/`, `embeddings-server/`, and `admin/`
- **Baseline config**: `.bandit` file with tuned severity levels

#### Results in v0.6.0

- 0 critical vulnerabilities found
- All findings are LOW severity (expected patterns like assert statements in production code, hardcoded test data)
- Configurable suppress list in `.bandit` prevents false positives from disrupting CI

### Checkov IaC Scanning (SEC-2)

#### What it does

Checkov scans Infrastructure-as-Code (Docker Compose, Kubernetes, Terraform) for security misconfigurations:
- Missing or weak health checks
- Insecure environment variable practices
- Inadequate resource limits
- Unrecommended base images

#### How it runs

- **Trigger**: Every push to any branch
- **Workflow**: `.github/workflows/security-checkov.yml`
- **Scope**: `docker-compose.yml`, `docker-compose.override.yml`, and all files in `deployment/` subdirectory

#### Results in v0.6.0

- Passed validation with all recommended controls in place
- Health checks are properly configured on all services
- Resource limits are defined for production services
- Restart policies prevent cascade failures

### Zizmor GitHub Actions Scanning (SEC-3)

#### What it does

Zizmor scans GitHub Actions workflows (`.github/workflows/*.yml`) for security issues:
- Unsafe use of secrets in log output
- Missing branch protection on workflow inputs
- Overly permissive `permissions:` directives
- Use of deprecated or vulnerable actions

#### How it runs

- **Trigger**: Every push to any branch
- **Workflow**: `.github/workflows/security-zizmor.yml`
- **Scope**: All `.yml` files in `.github/workflows/`

#### Results in v0.6.0

- All workflows pass zizmor validation
- Secrets are properly masked
- Workflow permissions are narrowly scoped

### Security Baseline Document (SEC-5)

The v0.6.0 security baseline (`docs/security/baseline-v0.6.0.md`) catalogs all findings across bandit, checkov, zizmor, and manual code review:

- **287 total findings triaged**
- **0 CRITICAL** vulnerabilities
- **36 HIGH** priority items deferred to v0.7.0 (e.g., TLS certificate validation, audit logging)
- **14 MEDIUM** priority items accepted with documented mitigations
- **237 LOW** priority items accepted (expected patterns, test-only code)

The baseline serves as the security roadmap and is updated with each release.

## Container Hardening

### Health Checks

Every container in the Docker Compose stack now includes a health check to verify that services are:
1. Running
2. Ready to accept connections
3. Responsive to queries

#### Health check configuration

Each service defines:
- **test command** — the probe command (e.g., `redis-cli ping`, `wget --spider`)
- **interval** — how often to run the probe (5–30 seconds depending on service)
- **timeout** — how long to wait for a response (5–30 seconds)
- **retries** — how many consecutive failures trigger "unhealthy" state
- **start_period** — grace period after container start before first probe (10–60 seconds)

#### Example: Redis health check

```yaml
redis:
  healthcheck:
    test: ["CMD", "redis-cli", "ping"]
    interval: 5s
    timeout: 15s
    retries: 1
  restart: unless-stopped
```

#### Service dependencies

Services that depend on others now use `condition: service_healthy` instead of just `service_started`, ensuring that dependent containers wait for their upstreams to be ready:

```yaml
document-indexer:
  depends_on:
    redis:
      condition: service_healthy
    rabbitmq:
      condition: service_healthy
```

### Resource Limits

All services have memory limits defined to prevent one service from consuming all host resources:

| Service | Memory Limit | Memory Reservation |
|---------|--------------|-------------------|
| Redis | 512 MB | — |
| RabbitMQ | 2 GB | — |
| SolrCloud nodes (×3) | 2 GB each | 1 GB each |
| Embeddings Server | 2 GB | — |
| Solr Search API | 512 MB | 256 MB |
| Streamlit Admin | 512 MB | 256 MB |
| Redis Commander | 256 MB | — |
| nginx | 256 MB | 128 MB |

### Restart Policies

Services are configured with production-appropriate restart strategies:

- **`restart: unless-stopped`** — automatically restart unless explicitly stopped by user
  - Used for core services (Redis, RabbitMQ, Solr, Embeddings Server)
  - Ensures availability after transient failures
- **`restart: on-failure`** — only restart if the container exits with a non-zero code
  - Used for document-indexer and document-lister (these should exit cleanly when stopped)
- **`restart: no`** — never restart (for one-off init containers like solr-init)

### Stop Grace Period

Each service specifies `stop_grace_period` to allow graceful shutdown before forced termination:

- **10 seconds** — for stateless services (nginx, redis-commander)
- **30 seconds** — for services with in-flight operations (Redis, Document Indexer)
- **60 seconds** — for distributed systems (Solr nodes, ZooKeeper)

Example:
```yaml
document-indexer:
  stop_grace_period: 30s
  restart: on-failure
```
