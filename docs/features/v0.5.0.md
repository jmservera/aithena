# v0.5.0 Feature Guide

Aithena v0.5.0 expands the Search experience with selectable search modes, semantic recommendations, and an embedded admin dashboard, while also hardening the ingestion pipeline and adding the first frontend component test suite. This guide describes the shipped behavior verified in the current source code.

## Feature summary

- **Search mode selector** for switching between keyword, semantic, and hybrid search.
- **Similar Books panel** that appears after opening a book and recommends semantically related titles.
- **Admin tab** that embeds the Streamlit operator dashboard inside the React app.
- **RabbitMQ startup hardening** with a pinned image version and a real health check grace period.
- **Pipeline startup ordering** that waits for RabbitMQ and Redis health before starting dependent services.
- **Language detection fix** that aligns Solr's detected-language field and adds folder-based language extraction.
- **Frontend test coverage** with Vitest + React Testing Library for the main search UI flows.

## Search mode selector

The Search page now exposes three mode buttons directly beside the search form:

- **Keyword** — the default mode. Uses the standard Solr keyword query path.
- **Semantic** — sends the query through the embeddings flow and runs vector similarity search.
- **Hybrid** — combines keyword and semantic results with reciprocal rank fusion (RRF).

### How it behaves

- The default mode is **keyword**.
- Clicking a mode button immediately updates the active mode and resets pagination back to page 1.
- Every search request includes the selected mode as the `mode` query parameter sent to `/v1/search`.
- After a search runs, the selected mode is shown again as a badge next to the result count.

### User-facing details

- Button labels are **Keyword**, **Semantic**, and **Hybrid**.
- The buttons include tooltips describing each mode as traditional keyword, vector-based semantic, or combined search.
- If a user switches to **semantic** or **hybrid** without entering a query, the UI shows an inline error instead of running an empty search.
- If the backend rejects a semantic or hybrid request with a `400`, the UI surfaces a friendly message that embeddings may not be indexed yet.

## Similar Books panel

The Similar Books feature is tied to the PDF opening flow.

### When it appears

- Run a search and open a result with **📄 Open PDF**.
- As soon as a result becomes the selected book, the page renders the **Similar Books** panel underneath the results area.
- The panel stays tied to the currently selected book in the PDF viewer.

### How it works

The panel calls:

- `/v1/books/{documentId}/similar?limit=5&min_score=0.0`

That shipped behavior means:

- the source book itself is excluded by the backend,
- up to **5** recommendations are returned,
- each recommendation includes title, author, optional year/category, and a similarity score,
- the UI formats the similarity score as a rounded percentage such as **91% match**.

### Loading, empty, and error states

The panel has explicit states in the shipped UI:

- **Loading** — shows “Loading similar books…” plus skeleton cards.
- **Empty** — shows “No similar books found”.
- **Error** — shows a friendly inline error telling the user to try another title in a moment.

### How users move through recommendations

- Each similar book is rendered as a button-like card.
- Clicking a similar book updates the selected document in the existing PDF viewer flow.
- The hook caches fetched similar books, so selecting one can immediately swap the viewer without needing the original search results to contain that book.

## Admin tab

v0.5.0 adds a dedicated **🛠️ Admin** tab to the main navigation.

### What it renders

The React page is intentionally simple:

- route: `/admin`
- content: a full-page iframe
- iframe source: `/admin/streamlit/`

The iframe is sandboxed with:

- `allow-same-origin`
- `allow-scripts`
- `allow-forms`
- `allow-popups`

### What users see inside it

The embedded Streamlit dashboard currently shows:

- **Total Documents**, **Queued**, **Processed**, and **Failed** counters from Redis
- **RabbitMQ Queue** metrics for ready, unacked, and total messages
- sidebar access to **Document Manager**, where operators can inspect queued/processed/failed documents and trigger requeue or clear actions

This gives operators an admin surface without leaving the main React application shell.

## RabbitMQ startup hardening

The Compose stack now hardens RabbitMQ startup in two ways:

1. **Version pin** — the image is pinned to `rabbitmq:3.13-management` instead of floating on the generic management tag.
2. **Health check tuning** — the service now uses:
   - `rabbitmqctl ping`
   - `interval: 10s`
   - `timeout: 30s`
   - `retries: 12`
   - `start_period: 30s`

The service also sets `RABBITMQ_VM_MEMORY_HIGH_WATERMARK=0.6` alongside the existing long consumer timeout.

### Why it matters

This reduces false startup failures and gives the queue enough time to become genuinely ready before the rest of the ingestion pipeline depends on it.

## Pipeline startup ordering fix

The v0.5.0 Compose changes also tighten dependency ordering.

### Services now waiting on healthy dependencies

- `document-lister` waits for **RabbitMQ** and **Redis** with `condition: service_healthy`
- `document-indexer` waits for **RabbitMQ** and **Redis** with `condition: service_healthy`
- `streamlit-admin` waits for **RabbitMQ** and **Redis** with `condition: service_healthy`
- `redis-commander` waits for **Redis** health
- `nginx` waits for **RabbitMQ** health before proxying the admin surfaces

### Release impact

This removes a common race at startup where the lister or indexer could start before RabbitMQ was actually ready to accept work.

## Language detection fix

The v0.5.0 language fix has two parts.

### 1. Solr detected-language field alignment

The Solr `langid` update processor now writes detected values into:

- `language_detected_s`

This matches the field used elsewhere in the schema and search responses.

### 2. Folder-path language extraction in the indexer

The document indexer now inspects folder names and treats known ISO 639-1 language-code segments as language hints.

Examples of shipped cases include paths such as:

- `ca/Novel·la/Víctor Català/Solitud.pdf`
- `es/Poesia/García Lorca - Romancero gitano (1928).pdf`
- `fr/Villon - Ballade (1490).pdf`
- `la/Cicero - De re publica.pdf`

When a language folder is present, the indexer stores that value in `language_s`. The search API then reads language data with a `language_detected_s` first / `language_s` fallback for filters, result normalization, and search facets.

### Why a full reindex is required

Documents indexed before this fix can have inconsistent language metadata because:

- Solr's automatic language detector was writing to the wrong field name, and
- folder-based language hints did not exist yet.

A full reindex ensures the whole collection is rebuilt with the corrected field mapping and folder-derived language metadata.

## Frontend tests

v0.5.0 introduces the first colocated frontend component tests in `aithena-ui`.

### Test stack

- **Vitest** as the test runner
- **jsdom** as the browser-like environment
- **@testing-library/react** for rendering and querying components
- **@testing-library/user-event** for realistic interaction tests
- **@testing-library/jest-dom** matchers via `vitest.setup.ts`

### What is covered

The shipped suites cover four UI areas:

- **SearchPage** — empty state, result rendering, API failure handling, and the PDF → Similar Books flow
- **SimilarBooks** — loading, success, empty, click, and error behavior
- **FacetPanel** — rendering, counts, select/deselect behavior, and hidden empty groups
- **PdfViewer** — dialog rendering, close controls, iframe URL handling, page anchors, and missing-document fallback state

These tests do not replace end-to-end validation, but they do give the release a real component-level safety net that did not exist in v0.4.0.
