# v0.11.0 — Local Authentication and First-Run Installer

Aithena v0.11.0 introduces a complete local authentication flow for the standard nginx entrypoint, plus an installer that turns first-run configuration into a guided, repeatable setup step.

## Release status

The milestone implementation is complete and documented. See the [release notes](../release-notes-v0.11.0.md) for the user-facing summary and the [test report](../test-report-v0.11.0.md) for the current release-gate status.

## Shipped features

### Local auth backend (#250, #251)

Aithena now ships with a built-in authentication module in `solr-search`.

Key capabilities:

- SQLite-backed local user store
- Argon2id password hashing
- JWT login and validation flow
- Bearer-token and cookie-based session support
- Auth middleware for protected backend routes

**User / operator benefit:** Aithena can now protect sensitive search, document, and admin surfaces without depending on an external identity provider.

---

### Login UX and protected React routes (#252)

The React UI now includes a dedicated login experience and session-aware routing.

What changed:

- `AuthContext` tracks session state in the browser
- `LoginPage` provides the interactive sign-in flow
- `ProtectedRoute` redirects unauthenticated users to `/login`
- Authenticated requests automatically include bearer credentials when needed

**User benefit:** users get a clear sign-in flow instead of opaque API failures, and protected pages now fail closed.

---

### nginx auth gating for APIs, documents, and admin tools (#253, #254)

The nginx gateway now validates sessions before serving protected content.

Protected surfaces through the standard nginx entrypoint:

- `/v1/*` application APIs
- `/documents/*` downloads
- `/admin/*` browser-facing admin tools

Behavior highlights:

- API clients receive `401` responses when unauthenticated
- Browser users are redirected to `/login` for protected interactive pages
- Public operational endpoints needed for health/version discovery remain accessible

**User / operator benefit:** the primary deployment surface now enforces a single access-control layer for both browser traffic and API consumers.

---

### Installer-driven first-run setup (#255, #256)

A new installer CLI is now the default way to prepare an Aithena deployment.

What it does:

- writes `.env`
- generates a JWT signing secret
- creates the auth database directory for the Docker bind mount
- seeds the initial admin user
- supports idempotent re-runs and reset flows

Typical command:

```bash
python3 -m installer
```

**Operator benefit:** new environments can be bootstrapped consistently, and rerunning setup no longer depends on hand-editing secrets or auth paths.

---

### Expanded auth and installer validation (#257)

The milestone also ships with broader automated coverage for the new flow.

Current highlighted counts:

- **140 backend tests** in `solr-search`
- **83 frontend tests** in `aithena-ui`

Coverage includes login behavior, protected route handling, installer flows, and auth-dependent UI behavior.
