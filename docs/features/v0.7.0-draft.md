# v0.7.0 Changelog (Draft)

Aithena v0.7.0 introduces **semantic versioning infrastructure** for all services, **version endpoints** for monitoring and debugging, **system status visibility** in the admin UI, and **CI/CD automation** for container orchestration. This document outlines the planned features for the upcoming v0.7.0 release.

## Planning status

v0.7.0 is currently in development. Issues #199–#204 are assigned and under active implementation.

**Estimated release**: Q2 2026

## Planned features

### Semantic Versioning Infrastructure (#199)

#### What it does

Implements consistent version tracking across all services without manual synchronization.

#### How it works

- Each service (`solr-search`, `document-indexer`, `document-lister`, `embeddings-server`, `admin`) reads version from:
  - Git tags (in development)
  - Container image labels (in production)
  - Hardcoded fallback (pre-release builds)
- Single source of truth: `version.txt` in project root
- CI automatically bumps version on tagged commits

#### User benefit

- Ops teams can quickly verify that all services are at the same version
- Debugging version mismatches becomes straightforward
- Deployment pipelines can enforce version consistency

### Version Endpoints for All Services (#200)

#### What it does

Exposes `/version` endpoint on every service for querying software version and build metadata.

#### Endpoint specification

**GET /version**

Response (200 OK):
```json
{
  "service": "solr-search",
  "version": "0.7.0",
  "build_time": "2026-03-20T14:30:00Z",
  "git_commit": "a1b2c3d4",
  "git_branch": "main",
  "python_version": "3.11.7"
}
```

#### Services covered

- `solr-search` (FastAPI)
- `document-indexer` (Python service)
- `document-lister` (Python service)
- `embeddings-server` (FastAPI)
- `admin` (Streamlit app)
- `aithena-ui` (React app via `/api/version` proxy through nginx)

#### User benefit

- Monitoring systems can poll all endpoints and detect version skew
- Debugging support: support team can ask users "what version are you running?"
- Health dashboards can include version info alongside uptime

### Version Display in UI (#201)

#### What it does

Shows the running Aithena version in the frontend UI (footer or header).

#### What users see

- **Footer version badge** — small text showing "Aithena v0.7.0" in the bottom-right corner
- **Admin system status** — version of each service listed in the System Status panel (see #203)
- **Hover tooltip** — when hovering over version, shows full metadata (commit, build time)

#### Behavior

- Version is fetched once on app load from `/api/version` (proxied through nginx)
- No refresh on navigation (static for the session)
- If `/api/version` fails, version displays as "unknown" without breaking the UI

#### User benefit

- Users can quickly confirm they're running the latest version
- Screenshot evidence of which version they tested
- Reassurance that the running app matches expected version

### Admin Containers Endpoint (#202)

#### What it does

Exposes a read-only endpoint for the admin dashboard to query Docker container metadata without needing Docker socket access.

#### Endpoint specification

**GET /admin/api/containers**

Response (200 OK):
```json
{
  "containers": [
    {
      "name": "redis",
      "state": "running",
      "status": "Up 2 hours",
      "health": "healthy",
      "cpu_percent": 0.2,
      "memory_mb": 45,
      "memory_limit_mb": 512,
      "uptime_seconds": 7200
    },
    {
      "name": "solr-search",
      "state": "running",
      "status": "Up 2 hours",
      "health": "healthy",
      "cpu_percent": 1.5,
      "memory_mb": 320,
      "memory_limit_mb": 512,
      "uptime_seconds": 7200
    }
  ]
}
```

#### Security considerations

- Endpoint is served by the search API (solr-search) on localhost only
- Requires authentication if exposed externally (future work)
- Returns only non-sensitive metadata (no environment variables, secrets, or config)
- Operator must explicitly enable via environment variable `EXPOSE_CONTAINER_STATS=true`

#### How it works

- Backend queries Docker API (requires Docker socket volume mount in compose file)
- Data is cached with 10-second TTL to avoid hammering Docker API
- Available only in containerized deployments (not in local dev)

#### User benefit

- Operators can monitor all service health from the admin dashboard without SSH
- Real-time CPU and memory usage visible
- Quick identification of unhealthy or restarting services

### System Status Admin Page (#203)

#### What it does

Enhances the admin dashboard with a comprehensive System Status panel showing version, health, and resource info for all services.

#### What it shows

- **Version matrix** — table showing service name, version, and last update time
- **Health status** — summary of each service's health check state
- **Resource usage** — CPU and memory consumption
- **Uptime** — time since last restart for each service
- **Dependency graph** — visual representation of service startup order
- **Event log** — recent health check failures, restarts, version updates

#### Layout

- Streamlit multi-tab interface:
  - **Versions** tab — version matrix and update history
  - **Health** tab — health check statuses and history
  - **Resources** tab — CPU/memory graphs and alerts
  - **Logs** tab — recent system events

#### Auto-refresh

- Polls `/version` and `/admin/api/containers` every 30 seconds
- Highlights changed metrics in green (improvement) or red (degradation)
- Can be paused by operator for detailed inspection

#### User benefit

- Single pane of glass for system health
- Operators can diagnose version mismatches or unhealthy services in seconds
- Historical view helps identify patterns (e.g., memory leak, frequent restarts)

### CI/CD Automation (#204)

#### What it does

Automates version bumping, changelog generation, and container image tagging on release.

#### Release workflow

1. **Trigger**: Merge to main branch (or manual trigger)
2. **Version bump**: Determine new version using conventional commits
3. **Update artifacts**:
   - `version.txt` (new version)
   - `CHANGELOG.md` (auto-generated from commit messages)
   - Git tag (e.g., `v0.7.0`)
   - Container image tags (`:latest`, `:v0.7.0`)
4. **Build containers**: Rebuild images with new version labels
5. **Push artifacts**: Tag git commit, push to container registry
6. **Release notes**: Generate GitHub Release with changelog and artifacts

#### Supported workflows

- **Automatic versioning** (conventional commits) — reads commit messages, auto-increments version
- **Manual versioning** — release coordinator specifies version via GitHub Release creation
- **Pre-release tags** — support for `v0.7.0-rc1`, `v0.7.0-beta`, `v0.7.0-alpha`

## Rollback strategy

If v0.7.0 needs to be rolled back:

1. Previous container images remain available under `:v0.6.0` tag
2. Git history is preserved; checkout old tag: `git checkout v0.6.0`
3. No database migrations; version endpoints don't require schema changes
