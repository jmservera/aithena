# v0.4.0 Feature Guide

> Screenshots pending — capture when stack is running with `docker compose up`.

Aithena v0.4.0 ships a search-first reading experience for the indexed PDF library, plus two operational dashboards for health and collection reporting. This guide describes the behavior that is visible in the current source code and UI.

## Feature summary

- **Search page** for keyword search, sorting, paging, and result snippets.
- **Faceted filtering** by author, category, language, and year.
- **PDF viewer** that opens a result in an overlay and jumps to the matched page when page metadata is available.
- **Status tab** with indexing counts plus Solr, Redis, and RabbitMQ health.
- **Stats tab** with collection totals, page-count statistics, and facet breakdowns.

## Navigation

When the app opens, it redirects to the **Search** tab. The main navigation also exposes **Status** and **Stats**, which are documented below.

## Search page

The Search page is the main workspace for finding books.

![Search page placeholder](images/placeholder-search-page.png)

### How keyword search works

- The search box is designed for plain-language keywords.
- In the shipped UI, users search by entering a term and pressing **Search**.
- Results are fetched from the `/v1/search` API in **keyword** mode by default.
- The backend search logic is wired to Solr's edismax query parser and uses the indexed book fields behind the scenes.

### What fields are searchable

The current Solr-backed search implementation is documented around these indexed fields:

- **Title** via `title_s` / `title_t`
- **Author** via `author_s` / `author_t`
- **Full text content** via `content`
- **Combined catch-all field** via `_text_`

In practice, that means users can search for:

- book titles
- author names
- words that appear inside the indexed PDF text

### What appears in each result

Each result card can show:

- title
- author
- year
- category
- language
- page count
- matched page range, when available
- relative file path
- up to three highlight snippets

### Search controls

After a query runs, the page also offers:

- **Sort** options: Relevance, Year (newest), Year (oldest), Title (A–Z), Author (A–Z)
- **Per page** options: 10, 20, or 50 results
- **Pagination** controls for moving between pages of results

## Faceted filtering

Facets appear in the left sidebar and narrow the current keyword search.

![Facet filtering placeholder](images/placeholder-facet-filtering.png)

### Available filters

v0.4.0 exposes these facet groups:

- **Author**
- **Category**
- **Language**
- **Year**

Each facet value shows a document count in parentheses.

### Filter behavior

- Each facet behaves like a single-select option in the current UI.
- Users can combine filters across different facet groups.
- Active filters appear above the results as removable chips.
- If more than one filter is active, a **Clear all** button is shown.
- Changing a filter resets the results list back to page 1.

## PDF viewer

From any result that includes a document URL, users can open the PDF directly from the results list.

![PDF viewer placeholder](images/placeholder-pdf-viewer.png)

### How it opens

- Click **📄 Open PDF** on a result card.
- The PDF opens in an overlay dialog inside the app.
- The viewer uses the document URL returned by the search API.
- If the PDF cannot load in the embedded frame, the dialog offers a fallback link to open it in a new browser tab.

### Page navigation from search results

If the indexed result contains matched page metadata, the viewer appends `#page=` to the PDF URL and opens on the first matched page.

That means users can jump straight from a search result to the most relevant page when the document includes:

- `page_start_i`
- `page_end_i`

The result card also shows **Found on page X** or **Found on pages X–Y** when that data exists.

### Closing the viewer

Users can close the overlay by:

- clicking the close button
- pressing the **Escape** key

## Status tab

The Status tab gives users a quick health view of the system and the indexing pipeline.

![Status tab placeholder](images/placeholder-status-tab.png)

### What it shows

The page has two sections:

#### 1. Indexing Progress

- **Discovered** — total documents seen in Redis tracking
- **Indexed** — documents marked as successfully indexed
- **Failed** — documents marked as failed
- **Pending** — documents still waiting or not fully processed

#### 2. Service Health

- **Solr** — status, live node count, and indexed document count
- **Redis** — up/down indicator
- **RabbitMQ** — up/down indicator

### Auto-refresh behavior

- The Status tab refreshes automatically every **10 seconds**.
- The page also shows a **last updated** time.
- If a refresh fails after a previous successful load, the last good data remains visible and the error is shown inline.

## Stats tab

The Stats tab provides a high-level summary of the indexed library.

![Stats tab placeholder](images/placeholder-stats-tab.png)

### What collection stats are displayed

The summary area shows:

- **Books indexed** total
- **Total pages** across the collection
- **Average** pages per book
- **Min** page count
- **Max** page count

### Facet breakdowns

Below the summary, the page displays tables for:

- **By Language**
- **By Author (top 20)**
- **By Year**
- **By Category**

Each table shows a value and its document count.

### Refresh behavior

Unlike the Status tab, the Stats tab loads its data once when the page opens. To see new totals after more books are indexed, refresh the page.
