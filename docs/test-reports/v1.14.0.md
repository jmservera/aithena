### Solr Bind-Mount Permissions Fix

**Change:** Modified Solr entrypoint to run as root, drop to `solr` user via `gosu`.

**Code Review:**
- ✅ `src/solr/entrypoint-sasl.sh` — Proper root startup + `gosu solr` invocation
- ✅ `docker-compose.yml` — Entrypoint configuration updated
- ✅ `docker-compose.prod.yml` — Production configuration synced
- ✅ Directory ownership logic correct (chown /var/solr to solr:solr)
- ✅ No security regression (gosu is standard privilege drop tool)

**Validation Required:**
```bash
# After deployment, verify Solr data directory is writable
docker compose exec solr ls -la /var/solr/data
# Expected: drwxr-xr-x solr solr
```

### Restore Script DRY_RUN Compliance

**Change:** Updated restore scripts to respect `DRY_RUN` environment variable in pre-flight checks.

**Code Review:**
- ✅ Pre-flight checks gated by `[ "$DRY_RUN" != "true" ]`
- ✅ Dry-run mode reports what would happen without executing
- ✅ Allows safe validation before committing restore operation

**Validation Required:**
```bash
# Test dry-run mode
DRY_RUN=true ./scripts/restore.sh
# Should print planned operations without modifying state

# Test actual restore
DRY_RUN=false ./scripts/restore.sh
# Should execute restore and modify state
```

## Recommendations

1. **Testing First:** Operators should execute all test suites in a staging environment before deploying to production.
2. **Bind-Mount Verification:** If using bind-mounted Solr volumes, verify permissions after deployment with `docker compose exec solr ls -la /var/solr/data`.
3. **Offline Deployments:** Test restore script with `DRY_RUN=true` before committing to actual restore.
4. **Gradual Rollout:** Deploy to staging first, monitor for 24-48 hours, then promote to production.
5. **Monitoring:** Ensure monitoring tools are configured for Solr, RabbitMQ, and embeddings-server errors.
