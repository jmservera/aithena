## Shipped Features & Fixes

### 1. User Document Collections (#655, #659, #661, #664, #670, #674, #591)

Personal bookshelves with full CRUD lifecycle:

- **API Endpoints:**
  - `POST /v1/collections` — Create new collection
  - `GET /v1/collections` — List user's collections
  - `GET /v1/collections/{id}` — Get collection details
  - `PUT /v1/collections/{id}` — Update collection name/description
  - `DELETE /v1/collections/{id}` — Delete collection (cascade delete items)
  - `POST /v1/collections/{id}/items` — Add documents to collection
  - `PUT /v1/collections/{id}/items/reorder` — Reorder items
  - `PUT /v1/collections/{id}/items/{item_id}` — Update item notes
  - `DELETE /v1/collections/{id}/items/{item_id}` — Remove item from collection

- **Database:** SQLite backend with indexed lookups for search performance
- **Access Control:** Per-user isolation; users can only access their own collections
- **Frontend:** Collections page with creation modal, detail view, item management
- **Search Integration:** Collection membership displayed as badges on search results

**Tests Defined:**
Test suites exist in the repository for API CRUD operations, authorization, form validation, and edge cases (in `src/solr-search/tests/test_collections*.py` and `src/aithena-ui/src/__tests__/Collections*.test.tsx`). See `docs/test-reports/v1.10.0-collections.md` for test inventory. Operators should execute these tests before deployment.

### 2. Book Metadata Editing (#681, #683, #686, #688, #691, #695, #697, #593)

Single and batch editing with atomic updates:

- **Single Document Edit Modal:** Pop-up editor for title, author, series, tags
- **Batch Edit Panel:** Selection toolbar with bulk update operations
- **Solr Atomic Updates:** Partial field updates without full re-index
- **Redis Overrides:** Server-side metadata overrides persisted for re-index safety
- **Re-index Safety:** Validation prevents stale metadata after re-index
- **Security Review:** Included in the v1.10.0 milestone scope (#695)

**Tests Defined:**
Test suites exist in the repository for form validation, atomic updates, Redis consistency, and authorization (in `src/solr-search/tests/test_metadata_edit*.py`, `src/solr-search/tests/test_batch_metadata_edit.py`, and `src/aithena-ui/src/__tests__/MetadataEditModal.test.tsx`). Operators should execute these tests before deployment.

### 5. Backup & Restore Infrastructure (#657, #660, #663, #665, #669, #672, #676, #680, #594)

Tiered backup strategy with disaster recovery tooling:

- **Backup Scripts:**
  - `backup-critical.sh` — Auth database, collections database, and secrets
  - `backup-high.sh` — Solr and ZooKeeper backup data
  - `backup-medium.sh` — Redis and RabbitMQ state

- **Restore Orchestrator:** Validates backup integrity, restores in correct order, post-restore verification
- **API Endpoints:**
  - `POST /v1/admin/backups` — Trigger backup
  - `GET /v1/admin/backups/status` — Check backup progress
  - `POST /v1/admin/backups/{backup_id}/restore` — Restore from backup
  - `GET /v1/admin/backups` — List available backups

**Tests Defined:**
Test suites exist in the repository for backup script correctness, restore orchestration, API integration, and admin UI (in `src/solr-search/tests/test_backup_service.py` and `tests/restore/`). Operators should execute these tests before deployment.

### 7. Semantic Index 502 Error Fix (#646, #704)

URI-too-large error in vector search queries resolved:

- **Root Cause:** Embedding vectors exceed HTTP URI length limit for some queries
- **Fix:** Switch to POST body for embeddings queries
- **Backward Compatibility:** GET `/v1/search` still works; fallback to POST internally

**Tests Defined:**
Test suites for this feature should be executed from the repository test directories. Operators should verify vector search correctness and edge cases before deployment.
