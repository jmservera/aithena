## Security Hardening Tests

### Nginx TLS / HSTS Header (N-1)

**Issue:** Nginx missing HSTS header for TLS deployments

**Changes:**
- HSTS header configuration defined in `docker-compose.ssl.yml` for TLS deployments
- Policy: `Strict-Transport-Security: max-age=31536000; includeSubDomains` (1-year, includes subdomains)
- Operators enable via SSL overlay for HTTPS deployments
- Header sent only when TLS is active

**Test Coverage Required:**
- TLS deployment: HSTS header present in HTTPS responses
- Header value: `max-age=31536000; includeSubDomains`
- Plain HTTP: No HSTS header (as expected)
- Browser enforces HTTPS on subsequent requests

**Status:** Configuration defined; awaiting TLS deployment validation.

### Redis ACLs (R-1, R-2)

**Issue:** Redis has no ACLs — dangerous commands available to all clients

**Changes:**
- Redis ACL enabled with restricted user for indexing services
- Indexer user permissions: GET, SET, DEL, LPUSH, RPOP, LRANGE, SCAN, MGET (and variations)
- Dangerous commands blocked: FLUSHDB, FLUSHALL, CONFIG, SHUTDOWN, SLAVEOF
- Credentials injected via `.env`

**Test Coverage Required:**
- `redis-cli -u redis://indexer_user:password@localhost PING` — connection succeeds
- `redis-cli -u redis://indexer_user:password@localhost GET key` — read works
- `redis-cli -u redis://indexer_user:password@localhost FLUSHDB` — returns error
- `redis-cli -u redis://indexer_user:password@localhost CONFIG SET` — returns error
- Indexing pipeline uses Redis without permission errors
- Services handle ACL errors gracefully

**Status:** Code changes identified; awaiting ACL validation and error handling testing.
