## Overview

Aithena v1.4.0 is a **major dependency upgrade release** that modernizes the technology stack across all services. This release brings significant performance improvements, security enhancements, and ecosystem alignment with latest LTS releases.

**Key Changes Summary:**
- Python 3.11 → 3.12 (all services)
- Node.js 20 → 22 LTS (frontend)
- React 18 → 19 (aithena-ui)
- RabbitMQ 3.13 → 4.0 LTS
- Updated Python package dependencies
- ESLint v8 → v9 with flat config
- i18n (internationalization) infrastructure via react-intl

**Upgrade Type:** Full stack dependency upgrade with some breaking changes. Read this guide thoroughly before upgrading.

## What's New

### Python 3.12 Upgrade

All Python services (solr-search, document-indexer, document-lister, embeddings-server, admin) now require **Python 3.12 or higher**.

**Benefits:**
- Improved performance (3-10% faster in microbenchmarks)
- Better error messages with exception groups and enhanced traceback formatting
- Per-Interpreter GIL (Global Interpreter Lock) isolation for true parallelism
- Type system improvements and faster type checking
- Latest security patches and 5+ years of support (October 2028)

**Docker Images Updated:**
- `python:3.12-slim` (solr-search, document-lister)
- `python:3.12-alpine` (document-indexer)
- `python:3.12-slim-bookworm` (embeddings-server)

### Node.js 22 LTS Upgrade

Frontend build environment and tooling now use **Node.js 22 LTS**.

**Benefits:**
- Significant JavaScript engine performance improvements
- Better module resolution and ESM support
- Latest npm features (npm 10+)
- Extended support until April 2027

**Frontend Stack Updates:**
- React 19 with improved rendering and server component support
- Vite 8+ for faster builds
- ESLint 9 with flat config (breaking change — see below)

### React 19 Migration

The aithena-ui frontend now targets **React 19**.

**Key Changes:**
- Improved automatic batching for state updates
- New `useActionState` hook for form handling
- Simplified `<form>` action handling
- Better TypeScript support and inference
- Removed legacy StrictMode warnings in production

**Compatibility:** React 19 is a major release. Plugins and custom hooks may require updates.

### RabbitMQ 4.0 LTS

Message broker upgraded to **RabbitMQ 4.0 LTS** (from 3.13).

**Behavioral Changes:**
- Stream queue support by default (replaces AMQP queue legacy features)
- Improved message persistence and replication
- Updated management API endpoints
- Backwards compatible with existing AMQP queue declarations

### Enhanced Dependencies

**Python Packages Updated:**
- FastAPI: Latest compatible version
- Pydantic: v2.x with validation improvements
- PyJWT: Replaces python-jose for JWT handling (see breaking changes)
- Solr client libraries: Latest versions

**Frontend Dependencies:**
- React 19
- React Router v6 (latest)
- Vite plugins: Latest compatible versions

### i18n Infrastructure

Internationalization infrastructure added using **react-intl**.

**Details:** See `.squad/decisions.md` for i18n framework decision. Locale fallback: English (en-US) is default.

## Breaking Changes & Migration Notes

### 1. ESLint Configuration (Frontend)

**Change:** ESLint v8 → v9 with flat config format.

**Impact:** `.eslintrc.json` replaced with `eslint.config.js` (flat config format).

**Migration Steps:**
```bash
# Already applied in codebase
# If you have local ESLint configs:
1. Delete .eslintrc.json if present
2. Use eslint.config.js (already committed)
3. Run: npm run lint
```

### 2. PyJWT Migration (Backend Services)

**Change:** `python-jose` → `PyJWT` for JWT handling.

**Impact:** JWT token generation/validation now uses `PyJWT` instead of python-jose.

**Migration Steps:**
```bash
# Already applied in all Python services
# In custom JWT code, if any:
import jwt  # from PyJWT

# Replace python-jose:
# from jose import jwt  # OLD
# with:
from jwt import encode, decode  # NEW (from PyJWT)

# Token claims remain compatible — no data migration needed
```

**Location:** See `src/solr-search/app/api/auth.py` for reference implementation.

### 3. React Component Updates (Frontend)

**Change:** React 19 removes some legacy APIs; component API changes.

**Impact:** Components using deprecated React patterns may require updates.

**Common Updates Needed:**
- Replace `<form onSubmit={...}>` with `<form action={...}>` for native form handling
- Update refs from `useRef(null)` patterns if using deprecated createPortal APIs
- Review any custom hooks that rely on legacy React internals

**Testing:** See [Testing & Validation](#testing--validation) section.

### 4. Admin Dashboard (Streamlit Removed in v1.3+)

**Note:** Admin dashboard is now a **native React component**, not Streamlit. Ensure you're familiar with the new React-based admin UI.

**Location:** `src/aithena-ui/src/pages/AdminDashboard.tsx`

## Upgrade Procedure

### Prerequisites

1. **Backup Current Data:**
   ```bash
   docker compose exec solr tar czf /backup/solr-indices-v1.3.0.tar.gz /var/solr/data
   docker compose exec redis redis-cli --rdb /backup/redis-dump-v1.3.0.rdb
   ```

2. **Verify Current Version:**
   ```bash
   cat VERSION  # Should show 1.3.0
   ```

3. **Review This Guide** thoroughly and test in a staging environment first.

### Step 1: Stop Services

```bash
# Pull latest dev branch
git checkout dev
git pull origin dev

# Verify you're on commit with v1.4.0 changes
git log --oneline | head -5

# Stop running containers
docker compose down
```

### Step 2: Update Codebase

```bash
# Update VERSION file (if not already updated)
echo "1.4.0" > VERSION

# Pull/merge latest code with all upgrades
git fetch origin
git checkout squad/353-upgrade-docs  # or main branch with v1.4.0 tag
```

### Step 3: Rebuild Docker Images

```bash
# Build all service images with Python 3.12, Node 22
./buildall.sh

# Watch for build errors — Python 3.12 or Node 22 specific issues
# Common issues: deprecated APIs, missing dependencies
```

### Step 4: Migrate Data (if needed)

**Most data migrations are automatic.** However, check:

1. **RabbitMQ Queue Compatibility:**
   ```bash
   # Old queues (v1.3.0) are auto-migrated to v1.4.0
   # Monitor logs for queue creation errors:
   docker compose logs rabbitmq | grep -i error
   ```

2. **Solr Index Compatibility:**
   ```bash
   # Solr 9.x indices are forward-compatible
   # No reindexing needed
   # Verify index health after startup:
   curl -s http://localhost:8983/solr/admin/cores?action=STATUS | jq '.status'
   ```

3. **Redis:**
   ```bash
   # Redis 7.x data is fully compatible
   # No migration needed
   ```

### Step 5: Start Services

```bash
# Start all services
docker compose up -d

# Monitor startup
docker compose logs -f

# Wait for all services to be healthy (2-3 minutes)
# Check health:
docker compose ps

# All services should show "healthy" or "up" status
```

### Step 6: Verify Upgrade

```bash
# Check service versions
curl -s http://localhost:8080/version  # solr-search
curl -s http://localhost:8085/version  # embeddings-server
curl -s http://localhost:5173          # aithena-ui (should load)

# Verify Solr is ready
curl -s http://localhost:8983/solr/aithena/select?q=*:*&rows=1

# Test a simple search
curl -s "http://localhost:8080/api/search?q=test&limit=5"

# Check admin dashboard loads
# Navigate to http://localhost (via nginx)
```

### Step 7: Run Regression Testing

**⚠️ TODO: Regression testing pending (issue #352)**

Once issue #352 is completed, run:
```bash
# Run end-to-end smoke tests
cd e2e && npm test
```

2. Filter by topic/genre
3. Upload a document to index
4. Run a semantic search
5. Check admin dashboard stats
6. Monitor logs for errors

## Rollback Procedure

If critical issues are found in v1.4.0, rollback to v1.3.0 immediately. Follow these steps:

### Automatic Rollback (Recommended)

```bash
# Stop v1.4.0 services
docker compose down

# Revert to v1.3.0 tag
git checkout v1.3.0
echo "1.3.0" > VERSION

# Rebuild with Python 3.11, Node 20 images
./buildall.sh

# Restore data from backup (if queues were modified)
docker compose exec rabbitmq rabbitmq-queues list  # Check queue state

# Start services
docker compose up -d

# Verify rollback
docker compose ps
curl -s http://localhost:8080/version | grep "1.3.0"
```

### Manual Data Rollback (if needed)

```bash
# If Solr indices were corrupted:
docker compose exec solr bash -c "rm -rf /var/solr/data/* && tar xzf /backup/solr-indices-v1.3.0.tar.gz -C /"

# If Redis state was corrupted:
docker compose down
docker volume rm aithena_redis_data  # Wipe Redis volume
docker compose up redis
# OR restore from backup:
docker compose exec redis redis-cli < backup/redis-dump-v1.3.0.rdb

# Restart all services
docker compose up -d
```

### Verify Rollback Success

```bash
# Check version endpoints report 1.3.0
curl -s http://localhost:8080/version | grep version
# Should output: {"version": "1.3.0"}

# Verify data integrity
curl -s "http://localhost:8080/api/search?q=*&limit=1"  # Should return books

# Check service logs for errors
docker compose logs --tail=50
```

### Communication & Resolution

1. **File an issue** with detailed logs if rollback was necessary
2. **Document what failed** (component, symptom, logs)
3. **Wait for fixes** in a patch release (e.g., v1.4.1)
4. **Re-test in staging** before attempting upgrade again

## Known Issues & Limitations

### 1. Regression Testing Pending (Issue #352)

**Status:** The comprehensive regression test suite for v1.4.0 is still in development (issue #352).

**Workaround:** Manual testing required (see [Step 7](#step-7-run-regression-testing) above).

**Impact:** Some edge cases may not be covered until #352 is complete. Plan accordingly.

### 2. React 19 Component Compatibility

**Issue:** Some older npm packages may not declare React 19 as compatible.

**Workaround:** 
```bash
npm install --save-peer-deps  # Allows peer dependency mismatches
```

**Resolution:** Update affected packages or open GitHub issues with package authors.

### 3. RabbitMQ Stream Queues

**Issue:** Legacy AMQP queues may show deprecation warnings (not errors).

**Workaround:** Warnings are cosmetic; functionality is unchanged.

**Timeline:** Classic AMQP queues remain fully supported in RabbitMQ 4.0.

## Support & Troubleshooting

### Common Issues

#### Issue: `ImportError: No module named X` (Python services)

**Cause:** Python dependency mismatch after 3.11 → 3.12 upgrade.

**Solution:**
```bash
# Rebuild containers (forces fresh dependency install)
docker compose down
docker rmi aithena-solr-search aithena-document-indexer aithena-embeddings-server
./buildall.sh
docker compose up -d
```

#### Issue: RabbitMQ fails to start with "incompatible Erlang version"

**Cause:** RabbitMQ 4.0 requires Erlang 26+.

**Solution:**
```bash
# Check RabbitMQ logs
docker compose logs rabbitmq | tail -30

# Rebuild (docker-compose.yml has correct image)
docker rmi aithena-rabbitmq
./buildall.sh
```

#### Issue: Solr takes >5 minutes to start or remains "unhealthy"

**Cause:** Solr 9.x is initializing indices (normal on first run).

**Solution:**
```bash
# Wait and monitor
docker compose logs -f solr

# Once you see "Solr started", wait another 30 seconds
# Check health:
curl -s http://localhost:8983/solr/admin/cores?action=STATUS | jq '.status'
```

#### Issue: Frontend shows blank page or "Cannot reach API"

**Cause:** Vite dev proxy or API connectivity issue.

**Solution:**
```bash
# Check if solr-search service is up
docker compose logs solr-search | tail -20

# Verify API is responding
curl -s http://localhost:8080/version

# If dev mode, check Vite proxy config in vite.config.ts:
cat src/aithena-ui/vite.config.ts | grep -A 5 "proxy"
```

### Logging & Diagnostics

```bash
# Tail logs from all services
docker compose logs -f

# Tail specific service
docker compose logs -f solr-search

# Check container health
docker compose ps

# Inspect network
docker network ls
docker network inspect aithena_default

# Python service version check
docker compose exec solr-search python -c "import sys; print(sys.version)"
# Should show: Python 3.12.x

# Node version check (in UI container or local)
npm -v  # Should be 10.x
node -v  # Should be v22.x
```

### When to Seek Support

1. **Build fails:** Check Docker images and network connectivity
2. **Services won't start:** Review container logs for dependency errors
3. **Data is corrupted:** Use rollback procedure and file an issue
4. **Performance degrades:** Profile with Python cProfile or Node Clinic
5. **Regressions in functionality:** File issue with reproduction steps
