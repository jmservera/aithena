## Context

Apply this skill when modifying search queries, adding filter queries, changing the Solr schema's vector fields, or implementing new search modes in aithena. The parent-chunk model is the foundation of hybrid search and the most common source of correctness bugs.

## Patterns

### 1. Understand the two-tier document model

**Parent documents** represent whole books:
- `id` = SHA-256 of file path (deterministic, deduplicated)
- Metadata fields: `title_s/t`, `author_s/t`, `year_i`, `page_count_i`, `file_path_s`, `folder_path_s`, `category_s`, `language_detected_s`
- NO embeddings on parent documents
- Used for: keyword search results, facet aggregation, metadata display

**Chunk documents** represent text fragments of books:
- `parent_id_s` links back to the parent document
- `chunk_text_t` holds the text fragment
- `embedding_v` (512-dim, HNSW cosine) for semantic search
- `chunk_index_i`, `page_start_i`, `page_end_i` for ordering
- Chunking: 400 words, 50-word overlap, page-aware boundaries

### 2. Apply EXCLUDE_CHUNKS_FQ correctly — the #1 correctness rule

```python
EXCLUDE_CHUNKS_FQ = "-parent_id_s:[* TO *]"
```

This filter query excludes chunk documents (which have `parent_id_s` set) from results.

- **✅ Apply to keyword search:** BM25 should return parent documents only
- **✅ Apply to the BM25 leg of hybrid search**
- **❌ NEVER apply to kNN queries:** kNN must search chunk documents (that's where embeddings live)
- **❌ NEVER apply globally to hybrid search:** it would kill the semantic leg

**Why this matters:** PR #701 nearly broke semantic search by applying this filter globally. If `EXCLUDE_CHUNKS_FQ` reaches the kNN query, it excludes all documents that have embeddings → zero semantic results.

### 3. Search mode implementations

**Keyword (BM25):**
```
q={user_query} via edismax (qf=_text_, pf=title_t^2)
fq=EXCLUDE_CHUNKS_FQ  (parent docs only)
+ facets + highlighting
```

**Semantic (kNN):**
```
kNN query on embedding_v field (k=10 default)
NO EXCLUDE_CHUNKS_FQ  (must hit chunks)
Collapse results by parent_id_s → return parent metadata
No facets or highlighting (not meaningful for vector results)
```

**Hybrid (RRF):**
```
Parallel execution:
  Leg 1: BM25 with EXCLUDE_CHUNKS_FQ (parent docs)
  Leg 2: kNN without EXCLUDE_CHUNKS_FQ (chunk docs, collapsed)
Merge: Reciprocal Rank Fusion (k=60)
Result: unified ranked list with combined scores
```

### 4. Embedding pipeline

- Model: `distiluse-base-multilingual-cased-v2` (512 dimensions)
- Embeddings generated by `embeddings-server` at `POST /v1/embeddings/`
- Book-level embedding (`book_embedding`): computed during indexing, used for similar-books
- Chunk-level embedding (`embedding_v`): computed per chunk during indexing, used for search
- Query embedding: computed at search time by `solr-search` calling embeddings-server
- Empty query → 400 error (can't embed empty string; this is intentional)

### 5. Filter queries work across all modes

All search modes support the same filter query parameters:
- `fq_author`, `fq_category`, `fq_language`, `fq_year`
- These filter on parent document fields and are safe to apply to all query types
- For kNN, filters are applied post-kNN (Solr applies fq after vector scoring)

## Anti-Patterns

- **Don't apply `EXCLUDE_CHUNKS_FQ` to kNN queries** — this kills semantic search entirely
- **Don't assume embeddings exist on parent documents** — they live on chunks only
- **Don't add new `fq` parameters without testing all three search modes** — a filter safe for BM25 may break kNN
- **Don't skip the parent_id_s collapse step in semantic results** — users expect book-level results, not chunk fragments
- **Don't compute embeddings synchronously during search** — embedding generation can take >1s; solr-search manages this with configurable timeout (EMBEDDINGS_TIMEOUT=120s)
