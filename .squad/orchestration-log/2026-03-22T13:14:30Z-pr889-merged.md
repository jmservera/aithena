## Summary

Implemented operational tooling for test corpus indexing through dual pipelines and collection verification.

### Deliverables

- **`scripts/index_test_corpus.py`** — Publishes document file paths to RabbitMQ fanout exchange
  - Reuses same mechanism as `document-lister` (direct exchange publishing)
  - Accepts file paths from CLI or stdin
  - Safe to re-run; document-indexer idempotent via file path SHA-256 as Solr unique key

- **`scripts/verify_collections.py`** — Verification harness for dual Solr collections
  - Parent document count parity check
  - Document ID set equality validation
  - Embedding dimensionality sampling (1 sample per collection, not exhaustive)
  - Optional `--json` output for CI integration and exit code checking
  - Supports `--collection books` and `--collection books_e5base` flags

### Design Decisions

- **Scripts in `scripts/` (not in service)** — Operational tooling for multi-service orchestration
- **Idempotency via SHA-256 key** — Re-publishing same paths is safe; Solr overwrites on duplicate ID
- **Sampling dimensionality** — 1 sample per collection sufficient; full scan unnecessary and expensive (wrong dimensions fail at indexing time)
- **Empty collections pass checks** — Nothing to verify yet

### Integration

- No changes needed to document-indexer or document-lister (scripts use same exchange/queue pattern)
- Requires `pika` + `requests` (already available in dev containers)
- Lambert (tester) can integrate `verify_collections.py` into CI

- Verified against mock RabbitMQ and Solr instances
