## Summary

Implemented temporary A/B performance metrics collection for latency and resource monitoring.

### Deliverables

- **`GET /v1/admin/metrics`** — Retrieve rolling-window latency percentiles per collection and search mode
  - JSON response with p50, p95, p99 latencies (seconds)
  - Per-collection stats (`books`, `books_e5base`)
  - Per-search-mode breakdown (keyword, semantic, hybrid)
  - Requires admin auth (X-API-Key)

- **`POST /v1/admin/metrics/reset`** — Clear metrics store
  - Resets all latency samples
  - Requires admin auth
  - Useful for controlled test windows

- **`PerfMetricsStore`** class — Standalone in-memory rolling-window store
  - `threading.Lock` + `defaultdict` + `TimedSample` dataclass
  - Bounded memory via window pruning
  - No external dependencies (Prometheus, Redis, etc.)

### Design Decisions

- **In-memory store over Prometheus** — Existing `MetricsRegistry` is exposition-format only; refactoring for per-collection percentiles costly. This is temporary A/B tooling, not production monitoring.
- **Internal timing keys in search response** — `_solr_latency_s` + `_embedding_latency_s` returned in response dict, stripped by caller before client response. Underscore prefix signals internal use; avoids function signature changes.
- **Admin-auth on metrics endpoints** — Consistent with other `/v1/admin/*` endpoints. Performance data can reveal usage patterns; reset should be controlled.

### Integration

- Metrics recorded on every search request across all three search modes
- Rolling window (time-based pruning) keeps memory bounded
- Compatible with Ash's benchmark script (P2-2) for post-run evaluation
- Can be integrated into CI dashboards
