# Parker: Issue #1287 — Solr auth bootstrap: explicit role assignment

**Timestamp:** 2026-03-29T10:10:00Z  
**Status:** ✅ Complete  
**Tests:** 1026/1026 passing

## Summary

Fixed Solr's auth bootstrap entrypoint to explicitly assign roles after `solr auth enable`. Corrected readonly role from "search" (which doesn't exist) to "readonly". Added Solr credential generation to the installer's credential pipeline.

## Changes

1. **docker-compose.yml** — Updated solr-init entrypoint:
   - After `solr auth enable`, call `set-user-role` to assign `["admin"]` to admin user
   - Assign `["readonly"]` (not `["search"]`) to readonly user

2. **docker/compose.prod.yml** — Applied same pattern

3. **installer/** — Added Solr credential generation:
   - `build_env_values()` now generates: `SOLR_ADMIN_USER`, `SOLR_ADMIN_PASS`, `SOLR_READONLY_USER`, `SOLR_READONLY_PASS`
   - Production deployments no longer use hardcoded default passwords
   - Credentials rotate on next installer run

## Impact

- All RBAC-gated Solr operations (collection-admin-edit, etc.) now work
- Production deployments use generated secrets instead of defaults
- Team members modifying solr-init must ensure role assignments match `security.json` permission names

## Architecture Notes

Solr auth lifecycle: `solr auth enable` (create user) → `set-user-role` (assign permissions) → operations work. The RBAC setup is now explicit and traceable in the installer code.
