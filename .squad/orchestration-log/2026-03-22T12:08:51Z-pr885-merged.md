## Summary

Configured Docker Compose for the dual-model A/B test. Added `embeddings-server-e5` and `document-indexer-e5` services with appropriate resource limits, environment variables, and startup dependencies.

## Changes

### docker-compose.yml

#### embeddings-server-e5
- **Model:** `intfloat/e5-base-v2` (768D embeddings, 512-token context)
- **Memory:** 3GB limit / 2GB reservation (model size ~1.1GB + runtime headroom)
- **Environment:**
  - `MODEL_NAME=intfloat/e5-base-v2`
  - `HF_HUB_OFFLINE=1` (no runtime downloads)
- **Health check:** HTTP GET `/health`
- **Ports:** 8085 (internal; reverse proxy on public 8080)

#### document-indexer-e5
- **Image:** Reuses baseline `document-indexer` image (config-driven behavior)
- **Environment:**
  - `QUEUE_NAME=indexer_e5base`
  - `EXCHANGE_NAME=documents`
  - `SOLR_COLLECTION=books_e5base`
  - `EMBEDDINGS_HOST=embeddings-server-e5`
  - `EMBEDDINGS_PORT=8085`
  - `CHUNK_SIZE=300` (words)
  - `CHUNK_OVERLAP=50` (words, proportional scaling from 90/10 baseline)
- **Memory:** 512MB limit / 256MB reservation
- **Depends on:** `solr-init` (condition: service_completed_successfully)
  - Prevents indexer startup before target Solr collection exists

#### document-indexer (baseline)
- **Updated environment:**
  - `QUEUE_NAME=indexer_baseline` (was implicit; now explicit)
  - `EXCHANGE_NAME=documents`
  - `SOLR_COLLECTION=books` (unchanged)
- **Depends on:** `solr-init` (condition: service_completed_successfully)

### RabbitMQ Topology

No changes to `definitions.json` or RabbitMQ config. Fanout exchange (`documents`) and queue bindings (`indexer_baseline`, `indexer_e5base`) are declared dynamically by application code:
- **Publisher:** document-lister declares exchange at startup
- **Consumers:** Each indexer declares its queue and binding

Dynamic declaration is more resilient — queues exist only when consumers need them.
