## Task

Analyze multi-stage Docker approaches for embeddings-server Dockerfile to reduce ~4GB .venv COPY layer.

## Outcome

**Recommendation:** Approach 3 — BuildKit `--mount=from` bind mount

### Key Findings

- **Problem:** Current two-stage build creates ~4GB COPY layer containing all `.venv` (heavy deps duplicated from base image)
- **Root cause:** Docker `COPY` always writes full directory as new layer, even though base image already contains heavy packages
- **Solution:** Use BuildKit `--mount=from=ghcr.io/astral-sh/uv:latest` to transiently mount `uv` during build, install only delta deps (~200MB layer), then discard the mount
- **Result:** ~95% layer size reduction (4.1GB → 200MB when base cached) with no tools in runtime image

### Analysis Comparison

| Approach | Layer Size | uv in runtime | Complexity | Verdict |
|----------|-----------|--------------|-----------|---------|
| 1: In-place uv sync | ~200MB ✅ | ❌ Yes (~30MB) | Low | ✅ Good |
| 2: Multi-stage full COPY | ~4–8GB ❌ | ✅ No | Low | ❌ Don't use |
| 3: BuildKit --mount=from | ~200MB ✅ | ✅ No | Low | ✅✅ BEST |
| 4: Delta-only COPY | ~200MB ✅ | ✅ No | Very high | ❌ Too complex |
| 5: Strip + PYTHONPATH | ~1.1GB ⚠️ | ✅ No | Medium | ⚠️ Interim only |

### Prerequisites

Base image must be rebuilt to own `/app/.venv` with heavy deps pre-installed (torch, nvidia-*, triton, sentence-transformers). One-time change for both slim and openvino variants.

### Implementation Details

- `# syntax=docker/dockerfile:1` enables BuildKit syntax
- `RUN --mount=from=...` creates read-only bind mount for duration of RUN command only
- `uv sync --inexact --frozen --no-dev --no-install-project` installs only missing packages (delta)
- Removes Python symlink fix (venv created on same base)
- CI/CD: No changes needed (buildx-action already enables BuildKit)
