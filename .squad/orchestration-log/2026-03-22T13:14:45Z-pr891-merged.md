## Summary

Implemented internal comparison API for side-by-side search result evaluation across baseline and candidate collections.

### Deliverables

- **`GET /v1/search/compare`** — Compare search results across two Solr collections
  - Query parameter: `q` (required), `search_type` (keyword/semantic/hybrid), `limit` (default 10)
  - Baseline/candidate collections via server-side config (`COMPARISON_BASELINE_COLLECTION`, `COMPARISON_CANDIDATE_COLLECTION`)
  - Parallel queries via `ThreadPoolExecutor(max_workers=2)`; latency = max(baseline, candidate)
  - Returns both result sets + Jaccard-like overlap metric: |intersection| / max(|baseline|, |candidate|)
  - Endpoint hidden from OpenAPI docs (`include_in_schema=False`)

### Design Decisions

- **Internal endpoint** — Per PRD Phase 2 decision; no UI toggle. For benchmark script (P2-2) and future admin dashboard
- **Reuse search helpers** — Delegates to `_search_keyword`, `_search_semantic`, `_search_hybrid` for feature parity (degradation, circuit breakers, filters)
- **Parallel collection queries** — Avoids sum latency; enables responsive comparison workflow
- **Overlap metric uses `max` denominator** — |intersection| / max(|baseline|, |candidate|) prevents inflation when one side returns fewer results
- **Config via env vars** — Baseline/candidate not query parameters; prevents arbitrary collection comparisons
